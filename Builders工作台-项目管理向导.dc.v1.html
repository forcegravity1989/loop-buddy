从零开始 · 不是看板，是方法

用 AI 时代的方式，
一步步把一个项目的管理体系搭起来

我们会以一个真实样板项目 —— 「模型 API 服务 · 运维运营看板」 —— 从头走一遍：竞品对比 → 需求导入 → 洞察 → 北极星 → 引领 / 滞后指标 → 原型 → 进度管理。每一步都先讲「为什么」，再动手做。走完，你就拥有了一套可复制的项目管理方法，而不只是一块看板。

约 8 步 · 全程以样板项目示范
步骤 01 · 竞品对比

先回答「对标谁」

在写一行代码之前，先知道这个项目要对标谁、差距在哪。AI 时代信息获取几乎零成本——你不需要市场团队花数周访谈，自己用 agent 几小时就能扫描公开资料、提炼对比维度。

传统 → AI
市场 / 产品经理花数周访谈整理 → Builder 自己用 agent 扫描、提炼差距矩阵。
控制点
项目必须明确对标对象与差距，「缺口」就是你的差异化机会。

步骤 02 · 需求导入

把需求收敛成「问题」

不写 10 页 PRD。Anthropic 的做法是「原型即规格」——需求以一句问题陈述 + 几条用户故事的轻量形式导入，足够让 agent 直接开始做原型。

传统 → AI
10 页 PRD + 层层评审 → 一句问题陈述 + 用户故事 + 一个可验证的验收信号。
控制点
需求收敛为「可被验证的问题」，而不是一张功能清单。
问题陈述 · 一句话
用户故事
1
作为值班工程师，我希望在一个视图里看到可用性 / 延迟 / 成本 / 进行中事故，以便不再切 4 个工具。
2
作为值班工程师，我希望异常发生时拿到 agent 给的根因建议，以便更快定位。
3
作为负责人，我希望看到成本按模型 / 租户归因，以便判断哪个用法在烧钱。
验收信号
值班工程师能在单个视图、15 分钟内定位一次异常的根因。
步骤 03 · 洞察报告

提炼一个可被证伪的判断

把竞品与需求合成一个「洞察」：我们到底在解决什么本质问题、为什么是现在。洞察必须是一个判断（可能错），而不是一句正确的废话。

传统 → AI
厚厚的市场调研报告 → 一张可证伪的洞察卡 + 风险假设清单。
控制点
每个项目都有一个明确、可被证伪的核心假设。
核心洞察
运维的痛点不是「缺数据」，而是数据散落在 4 个工具里、且缺少按模型 / 租户的成本与质量归因。把它们收进一个值班视图，定位时间能从 38 分钟压到 15 分钟以内。
机会
现有工具要么只懂基础设施、要么只懂 LLM 调用，没人把两者放进同一个值班视图。这正是缺口。
风险假设 · 待证伪
假设「单视图 + 根因建议」真能缩短定位时间；若 agent 根因建议命中率太低，洞察不成立。
步骤 04 · 北极星指标

一个项目，只能有一个北极星

北极星是全队对齐的那一颗——OpenAI 用「这件事让我们更接近 AGI 吗」来取舍。它必须是用户价值导向、唯一、可量化，且从真实数据计算、难以人为修饰。

传统 → AI
一堆 KPI 各自为政 → 一个北极星统领，其余指标都为它服务。
控制点
有且仅有一个北极星；它衡量「用户得到的价值」，不是产出量。
本项目北极星
为什么它是个好北极星（可编辑）
✓ 用户价值导向
✓ 唯一
✓ 可量化

步骤 05 · 引领性指标

本周我能控制的先行动作

北极星是结果，引领指标是你本周能主动推动的先行动作。这是 AI 时代每周演进的抓手——每周设一次目标，让 agent loop 去推进。

三条铁律
可控（你本周推得动）· 可统计（系统自动出数）· 难造假（来源是真实日志，不是手填）。
控制点
引领指标必须同时满足「可控 / 可统计 / 难造假」，否则不要它。

步骤 06 · 滞后性指标

用来验证，不用来下周度命令

滞后指标反映结果，验证你的引领指标是否真的有效，但你无法直接操控它。它会滞后几周才动——所以它是「验收」，不是「本周行动目标」。

因果链
引领指标 驱动 北极星，滞后指标 验证 是否真的发生。
控制点
滞后指标只用于验证；若它不动，回头质疑引领指标选错了。

步骤 07 · 原型创建

步骤 08 · 过程 / 进度管理

每周可验证地演进

不是甘特图和状态周会，而是一个固定的周节奏 + 一个来自真实数据的健康信号。Cursor 当天就 ship，Anthropic 视野不超过 90 天、按周循环。

传统 → AI
甘特图 + 口头状态周会（易造假）→ 周度 shippable 增量 + 真实 telemetry。
控制点
每周演进由真实数据验证；人只在验收 / 质量门介入，不微管理 agent。

工作流 · 库与管理

工作流是独立管理的资产

一个工作流 = prompt + skills + agents + 阶段(phases) + 目标(验收指标) + loop(重试 / 循环次数)。每个都不一样（agent 定义不同）。点任意工作流查看其完整构成，或从下方各类 Hub 导入。

工作流库 · Workflow Library
工作流分两类——跑顺的「动态」会话沉淀为「静态」资产（含优化好的 agent 与 skills），越用越省。新项目从动态起步，老项目靠积累。
