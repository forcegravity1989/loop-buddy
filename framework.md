# Builders' Workbench —— 从传统项目管理到 AI 原生项目管理

> **目的**：不是为了限制 Builders 开发，而是用更好的管理方式辅助**每个项目健康地走完它的生命周期**。
> 本文档定义：传统 PM 的角色/流程 → 在 AI 原生构建下如何变化 → 新的「控制点」体系 → 落到 workbench 的数据模型与视图。

---

## 0. TL;DR

- 传统 PM 的厚重，本质是在为三种成本买单：**信息传递税、协调税、验证成本**。
- AI 同时打掉了两个前提——**执行成本 ≈ 0**、**信息获取成本 ≈ 0**——所以角色变少、单人职责边界变大，大量流程可以被 B 掉。
- 角色：约 **8 → 3**（Builder / 方向官 / 按需专家池）。流程：约 **10 → 精简主干**。
- 新管理的抓手不是「流程」而是 **6 + 1 个控制点**，原则是：**自动优先、按异常管理、少而锋利**。
- 你点名的五个要素全部落进控制点：竞品对比、需求导入、进度跟踪、北极星指标、滞后性指标。

---

## 1. 第一性原理：传统 PM 在为什么付费

先别盯着「流程」「角色」本身，先问它们**为什么存在**。几乎所有传统 PM 机器，都在为三种成本买单：

| 成本 | 含义 | 它催生了什么 |
|---|---|---|
| **信息传递税** | 专业分工导致 PM→设计→开发→QA 层层交接，每次交接信息都会失真 | 文档、评审、签字、对齐会 |
| **协调税** | 很多人做相互依赖的模块，需要同步谁在做什么、依赖与排期 | 项目经理、甘特图、站会、状态汇报 |
| **验证成本** | 执行慢、返工贵，所以要提前拦截、独立把关 | 独立 QA、阶段门(gate)、UAT |

**结论**：传统 PM 的厚重 = 「**执行成本高 + 信息靠人问**」这两个前提的产物。AI 恰好同时打掉了这两个前提——这是一切变化的根源。

## 2. 传统项目管理：角色与流程（基线）

> 你举例的「10 个流程、5 个角色」是示意。这里展开成一套行业可识别的真实版本（融合 PMBOK / Scrum / 经典软件交付）。

### 2.1 传统角色（约 8 个）

| 角色 | 核心职责 | 存在的根因 |
|---|---|---|
| 产品经理 (PM) | 需求、优先级、路线图 | 把「做什么」翻译成可执行规格 |
| 项目经理 / PMO / Scrum Master | 排期、协调、风险、状态汇报 | 多人多依赖的**协调税** |
| 设计师 (UX/UI) | 交互与视觉 | 专业分工 |
| 技术负责人 / 架构师 | 技术方案、任务拆解 | 把需求翻译成技术 |
| 开发工程师 | 实现 | 执行 |
| 测试 / QA | 质量验证 | 执行慢、返工贵 → 独立把关 |
| 数据分析师 | 取数、指标、报表 | 取数与分析有门槛 |
| 干系人 / Sponsor | 立项、审批、资源 | 决策与资源 |

### 2.2 传统流程（约 10 个 + 横切）

**主干（串行）**：
1. 立项 / 商业论证
2. 市场与竞品调研
3. 需求收集与分析（PRD）
4. 设计与技术评审
5. 计划与排期（WBS / 甘特 / Sprint planning）
6. 开发
7. 测试 / QA
8. 评审与验收（gate / UAT）
9. 发布 / 部署
10. 复盘与运营监控

**横切（贯穿全程）**：状态汇报、风险管理、变更管理、干系人沟通。

## 3. AI 原生构建带来的两个结构性变化

**变化 A：执行成本 → 趋近 0。**
一个 Builder + AI 可以横跨 设计 / 编码 / 测试 / 取数 / 文档。**单人职责边界变大**，交接次数骤减——很多「翻译/交接」型角色失去存在理由。

**变化 B：信息获取成本 → 趋近 0。**
过去要「找人问」才能拿到的状态、上下文、文档、「谁懂这块」，现在能被工具/AI 自动汇聚。**项目经理这一角色的大部分工作，本质就是这条信息税**——它最先被系统吸收。

### 3.1 前沿公司的公开可见模式

> 说明：以下来自公开演讲 / 博客 / 报道（截至训练资料），**并非这些公司的内部 playbook**；联网检索恢复后可替换为带链接的可核实版本。

- **Anthropic**：写作 / 文档驱动（设计文档、RFC 替代大量会议）、小团队、安全与 eval 驱动、重度 dogfood 自家工具（如 Claude Code）。
- **OpenAI**：小型「突击队」(small strike teams)、先发布再迭代、自下而上提案、相对扁平、研究与产品融合。
- **Cursor (Anysphere)**：极小团队撬动极高产出、工程师端到端 owner、重度 dogfood、高频发布。
- **通行趋势**：PM 从「写规格 + 协调」转向「**定方向 + 品味 + 设计 eval + 判断**」；PMO/项目经理职能被工具吸收；验证从人工 QA 转向自动 eval；瓶颈从「执行力」转移到「**意图清晰度与判断力**」。

## 4. 变化总表：谁被 B 掉、合并、升维

### 4.1 角色：约 8 → 3

| 传统角色 | 去向 | 归并到 |
|---|---|---|
| 项目经理 / PMO / Scrum Master | **被 B 掉** → 自动化 | 变成 workbench 系统本身 |
| 产品经理 | **升维** | → 方向官（定方向 / 品味 / eval / 优先级） |
| 技术负责人 / 架构师 | 多数并入 | → Builder；跨域架构按需顾问 |
| 设计师 | 部分并入 | → Builder（AI 生成迭代）；高端品味按需 |
| 开发工程师 | 转型 | → **Builder（端到端 owner）** |
| 测试 / QA | 转型 | → eval / 自动化；高风险面保留专家 |
| 数据分析师 | 部分并入 | → Builder 自助取数；深度分析按需 |
| 干系人 / Sponsor | 保留 | 通过 workbench 异常视图参与决策 |

**→ 三个核心角色：**
1. **Builder**：端到端 owner（设计→编码→测试→上线→度量）。
2. **方向官**（精简版 PM / 负责人）：定方向、问题、品味、eval、优先级。
3. **按需专家池**：设计 / 架构 / 安全 / 数据，被项目临时拉入。

> 若 Builder 含非工程造物者（增长 / 设计 / 内容），则「按需专家池」与控制点 5（质量门）需要按领域换成对应的质量标准（如品牌评审、法务合规、增长实验显著性）。

### 4.2 流程：约 10 → 精简主干

| 传统流程 | 去向 |
|---|---|
| 立项 / 商业论证 | 保留但极简 → **意图一页纸**（含北极星 + 成功标准） |
| 市场与竞品调研 | 转型 → **常态化、AI 持续监控**（非一次性阶段） |
| 需求 / PRD | 转型 → **意图捕获**（问题 + 成功标准 + eval），去重型文档 |
| 设计与技术评审 | 精简 → 1 次关键评审（品味 / 架构） |
| 计划与排期（WBS / 甘特） | **被 B 掉** → 短周期 + 实时信号 |
| 开发 | 保留，成本骤降 |
| 测试 / QA | 转型 → **持续自动评估**，砍掉独立 QA 周期 |
| 评审与验收门 | 精简 → 少而锋利的门 |
| 发布 / 部署 | 保留，门槛降低 → 高频发布 |
| 复盘与运营 | 保留并强化 → **滞后指标回收闭环** |
| 横切（状态汇报 / 风险） | **被 B 掉** → workbench 自动汇聚，砍掉状态会 |

## 5. 核心：新项目管理下的控制点

**设计哲学（= 你的目的）：控制点服务于「项目健康」，不服务于「控制人」。** 每个控制点满足三条：

1. **自动优先**——数据来自 git / 指标 / AI，而非让人填表。
2. **按异常管理**——绿色隐身，只浮出红 / 黄。
3. **少而锋利**——用最少的人工门，拦住真正会杀死项目的东西。

项目的典型「死法」 → 对应控制点（你点名的 5 要素已全部嵌入）：

### 控制点 1 · 北极星与意图对齐 〔北极星指标〕
- **拦的死法**：方向漂移 / 做了没人要的东西。
- **触发**：立项即声明——问题、目标用户、所服务的北极星、成功标准 / eval。**不对齐不开工。**
- **类型**：先行锚点。取代厚重 PRD / 立项书。

### 控制点 2 · 竞品与基线扫描 〔竞品对比〕
- **拦的死法**：被对手拉开身位而不自知。
- **触发**：AI 持续监控对手与标杆；对手有实质动作时主动告警。**常态化，而非一次性阶段。**
- **类型**：先行 · 常态。

### 控制点 3 · 需求导入与优先级 〔需求导入〕
- **拦的死法**：什么都做、什么都做不完（范围蔓延）。
- **触发**：用户反馈 / 数据 / 干系人诉求进**单一队列**，按北极星打分；设 **WIP 上限**。
- **类型**：先行 · 自动。

### 控制点 4 · 实时进度信号 〔进度跟踪〕
- **拦的死法**：悄无声息地停滞 / 隐藏的阻塞。
- **触发**：从 git / PR / 任务 / 预览部署**自动汇聚**；停滞 N 天、老化项、范围蔓延自动告警。
- **类型**：先行 · 自动。**这是替代项目经理的关键件。**

### 控制点 5 · 质量与评估门
- **拦的死法**：质量 / 安全回退。
- **触发**：eval + 自动化测试 + 安全检查，外加 **1 次关键人工评审**（品味 / 上线前）。砍掉 QA 周期与橡皮图章式的门。
- **类型**：门 · 人工 + 自动。

### 控制点 6 · 滞后指标回收 〔滞后性指标〕
- **拦的死法**：上线即遗忘，不知道到底有没有用。
- **触发**：上线后回看留存 / 收入 / 满意度是否真的变好（周级别）。**闭环。**
- **类型**：滞后 · 验证。

### 控制点 7（贯穿）· 组合健康度
- **拦的死法**：管理者要么瞎管、要么失明。
- **触发**：跨所有项目只看红 / 黄，**按异常管理**。这是「既保留监督、又不加协调税」的关键。
- **类型**：leadership 视图。

## 6. 先行指标 vs 滞后指标怎么配合

| | 先行指标（leading） | 滞后指标（lagging） |
|---|---|---|
| 作用 | **在途纠偏**（快反馈） | **事后验证真相**（慢反馈） |
| 例子 | 激活率、首值时间(TTFV)、eval 通过率、周期时间、上线首周使用率 | 留存、净收入留存、NPS、流失率 |
| 在控制点中 | CP1–CP4 | CP6 |

**北极星**通常是偏先行的「唯一重要指标」，但必须用滞后指标兜底校验，避免「北极星涨了、生意没变好」。

## 7. 让它不退化成「换皮官僚」的 6 条原则

1. **自动优先**：能从系统拿到的，绝不让人填表 / 开会汇报。
2. **按异常管理**：健康项目对管理者隐身。
3. **轻量意图 > 重型文档**：一页纸意图 + 原型 + eval，取代长 PRD。
4. **少而锋利的人工门**：全程 1–2 个真评审，而非 5 个签字。
5. **度量内建**：上线前就埋好指标，而非事后补。
6. **每个控制点对应一种「会杀死项目的风险」**：加控制点前先回答它拦的是哪种死法，否则不加。

## 8. 落到 workbench：数据模型与视图

### 8.1 数据模型（每个项目 = 一个「健康对象」）

```
Project {
  id, name, owner(Builder),
  northStar: { metric, aligned: bool },     // CP1
  intent: "一页纸：问题 + 成功标准 + eval",   // CP1
  competitors: [ { name, lastMove, flagged } ], // CP2
  backlog: { queued, wipLimit, wipNow },     // CP3
  signals: { progressPct, lastActivityDays, stalled, scopeCreep }, // CP4
  qualityGate: { evalPass, testsPass, humanReview: pending|passed }, // CP5
  lagging: { metric, baseline, current, collectAt },  // CP6
  health: green | yellow | red,              // 由上面自动推导
  flags: [ "进度停滞 4 天", "北极星未对齐", ... ]
}
```

`health` 由规则自动推导，例如：
- 任一关键告警（北极星未对齐 / 竞品实质动作 / 质量门未过）→ **red**
- 停滞 / WIP 超限 / 范围蔓延 → **yellow**
- 否则 → **green**

### 8.2 两个视图

- **方向官视图（Portfolio）**：默认落地页。顶部红 / 黄 / 绿计数 + 「只看红/黄」开关（按异常管理）；项目按健康度排序，红 / 黄置顶。点项目 → 详情。
- **Builder 视图**：选择某个 Builder → 我的项目 + 「今日需关注」（从控制点告警自动生成）。
- **项目详情**：6 个控制点的状态条（done / ok / active / alert / pending）+ 先行 vs 滞后指标面板。

## 9. 落地路线

| 阶段 | 内容 |
|---|---|
| **MVP（先打通自动信号）** | CP1 意图一页纸 + CP4 进度信号（接 git/PR）+ CP7 组合健康度视图 |
| **迭代 1** | CP3 需求队列 + WIP 上限；CP5 eval / 测试门接入 CI |
| **迭代 2** | CP2 竞品 AI 监控；CP6 滞后指标回收（接埋点 / 数仓） |
| **迭代 3** | Builder 视图个性化「今日需关注」；健康度推导规则可配置 |

## 10. 假设与待确认

- **假设**：Builder = 软件 / 产品工程师（端到端 owner）。若含非工程造物者，需调整专家池与 CP5 的质量标准（见 §4.1 注）。
- **待接数据源**：git/PR 平台、CI/eval、埋点 / 数仓、竞品监控源。
- **待定**：health 推导的具体阈值（停滞天数、WIP 上限）应按团队节奏校准。
