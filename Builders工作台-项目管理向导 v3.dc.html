从零开始 · 不是看板，是方法

用 AI 时代的方式，
一步步把一个项目的管理体系搭起来

我们会以一个真实样板项目 —— 「模型 API 服务 · 运维运营看板」 —— 从头走一遍：竞品洞察（证据→发现→洞察）→ 需求导入 → 北极星 → 引领 / 滞后指标 → 原型 → 进度管理。每一步都先讲「为什么」，再动手做。走完，你就拥有了一套可复制的项目管理方法，而不只是一块看板。

AI 时代 · Builders 模式
角色重定义（不是收敛成 1 个）
5 种角色原型：原型师 · 构建者 · 清理者 · 增长者 · 维护者。
角色 ≠ 职能、一人多角 —— 一个 Builder 往往同时是「原型师 + 构建者」。决定角色的是擅长做什么，不是部门。
流程精简（被划掉的，靠 agent 与真实数据替代）
② ③ PRD + 评审 → 原型即规格

步骤 01 · 竞品洞察

从证据，
一步步爬到判断

竞品分析不是「采集 → 报告」两步，而是一条 证据 → 发现 → 洞察 的链条。套用设计思维的研究菱形（Discover → Define）：先共情收集证据，再结构化摆事实，追问「所以呢」提炼发现，最后收敛成可证伪的判断。洞察因此是被推导出来的，不是被宣布的。

传统 → AI
市场团队数周访谈 + 厚报告 → Builder 用 agent 跑通采集与结构化，人只在「发现 → 洞察」这一跳介入。

发现 → 洞察 判断密度最高、最易翻车，由人把关；采集与结构化交给 agent。

05洞察 · 下判断— 产物：可证伪的洞察报告，站在发现之上
核心洞察 · POV
运维的痛点不是「缺数据」，而是数据散落在 4 个工具里、且缺少按模型 / 租户的成本与质量归因。把它们收进一个值班视图，定位时间能从 38 分钟压到 15 分钟以内。
机会
现有工具要么只懂基础设施、要么只懂 LLM 调用，没人把两者放进同一个值班视图。这正是缺口。
风险假设 · 待证伪
假设「单视图 + 根因建议」真能缩短定位时间；若 agent 根因建议命中率太低，洞察不成立。
步骤 02 · 需求导入
侧输入流与竞品发现做 reconcile

把需求收敛成「问题」

不写 10 页 PRD。Anthropic 的做法是「原型即规格」——需求以一句问题陈述 + 几条用户故事的轻量形式导入，足够让 agent 直接开始做原型。

传统 → AI
10 页 PRD + 层层评审 → 一句问题陈述 + 用户故事 + 一个可验证的验收信号。
控制点
需求收敛为「可被验证的问题」，而不是一张功能清单。
问题陈述 · 一句话
用户故事
1
作为值班工程师，我希望在一个视图里看到可用性 / 延迟 / 成本 / 进行中事故，以便不再切 4 个工具。
2
作为值班工程师，我希望异常发生时拿到 agent 给的根因建议，以便更快定位。
3
作为负责人，我希望看到成本按模型 / 租户归因，以便判断哪个用法在烧钱。
验收信号
值班工程师能在单个视图、15 分钟内定位一次异常的根因。
步骤 03 · 北极星指标

一个项目，只能有一个北极星

北极星是全队对齐的那一颗——OpenAI 用「这件事让我们更接近 AGI 吗」来取舍。它必须是用户价值导向、唯一、可量化，且从真实数据计算、难以人为修饰。

传统 → AI
一堆 KPI 各自为政 → 一个北极星统领，其余指标都为它服务。
控制点
有且仅有一个北极星；它衡量「用户得到的价值」，不是产出量。

步骤 04 · 引领性指标

本周我能控制的先行动作

北极星是结果，引领指标是你本周能主动推动的先行动作。这是 AI 时代每周演进的抓手——每周设一次目标，让 agent loop 去推进。

三条铁律
可控（你本周推得动）· 可统计（系统自动出数）· 难造假（来源是真实日志，不是手填）。
控制点
引领指标必须同时满足「可控 / 可统计 / 难造假」，否则不要它。

步骤 05 · 滞后性指标

用来验证，不用来下周度命令

滞后指标反映结果，验证你的引领指标是否真的有效，但你无法直接操控它。它会滞后几周才动——所以它是「验收」，不是「本周行动目标」。

因果链
引领指标 驱动 北极星，滞后指标 验证 是否真的发生。
控制点
滞后指标只用于验证；若它不动，回头质疑引领指标选错了。

步骤 07 · 过程 / 进度管理

看板是观测，
每周还得定目标

进度看板是客观观测——它只告诉你「现在是多少」。但按周推进，还要主动回答看板答不了的问题：根据上周的真实数据 + 本周要交付的特性，本周目标应该定成多少。这一步是手动的，也是「计划」的核心。

一个例子
上周「服务停机」3 次。本周上线了熔断与自动回滚，于是把本周目标从 3 次手动改为 0 次——目标的变化由数据和已交付的特性推导，而不是拍脑袋。
观测 ↔ 计划
看板回答「是多少」（观测）；周一复盘回答「该是多少」（计划）。两者闭环，才不是只看不动的甘特图。
控制点
每个本周目标都要挂一条依据（上周实际 + 本周交付的特性）。改目标可以，但必须说得出为什么。
本周节奏 · 一个闭环（观测 → 复盘 → 定目标 → 执行 → 回到观测）
周一 · 复盘
看上周目标 vs 实际
哪条达成、哪条没达成，看板直接给数。
周一 · 定目标
手动设本周目标
据上周实际 + 本周交付的特性推导，每条挂依据。
周中 · 执行
agent loop 推进
放手让 agent 跑，人只审关键逻辑与验收。
周五 · 观测
真实数据出健康信号
达成了吗？北极星在动吗？回到周一复盘。
↻ 每周一循环一次 · 上一圈的「观测」就是这一圈「复盘」的输入
