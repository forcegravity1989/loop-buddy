# 02 基石证伪：反向拆解「问题处理」Loop → 可复用模板

> R2 的靶子。MVP 全压在一个赌注上：**能否把现成 Loop 反向拆成「真正可复用」的模板（货架）。** 拆不出 → 工厂是 PPT、MVP 无地基。本文蓝队真拆一次，红队来证伪。**不空谈，判具体产物。**

---

## 1. 被拆对象：问题处理 Loop（取前一版已成形的具体结构）
- **流水线（7 步，领域具体）**：受理 → 复现 → 定位 → 修复 → 自测 → 评审 → 合并
- **Team（5 角色）**：主控(编排) / 受理 / 复现 / 开发 / 评审
- **验收**：那条「复现时写下、会失败的测试」转绿 + 评审通过；改到敏感面(登录)→ 转人工
- **Goal**：一个上报的 bug 被修好并合并，由「失败测试转绿」客观验证

## 2. 蓝队主张：可复用单元 ≠ 那 7 个领域步骤，而是「闭合任务模式 + 验收先行」
把领域词(复现/定位/修复)抽掉，剩下的骨架对任何**闭合任务**都成立：

### 2a. Workflow 模板 ——「闭合任务流水线」(6 抽象阶段)
| 阶段 | 动作 | 对应原 7 步 |
|---|---|---|
| ① 界定 | 确认任务真实、在范围内，写出一句话任务陈述 | 受理 |
| ② **建立验收（关键）** | **在动手前**写出一条**可自检的验收断言** | 复现(写失败测试) |
| ③ 执行 | 干活直到满足验收断言 | 定位+修复 |
| ④ 自检（循环点） | 跑验收断言；不达标 → 回 ③。**这就是 Loop 的「循环」** | 自测 |
| ⑤ 独立评审 | 另一个 agent（独立上下文）查质量 + 是否碰敏感边界 | 评审 |
| ⑥ 交付闸 | 验收过 且 无敏感边界 → 自动交付；否则 → 人工审批 | 合并(+登录转人工) |

### 2b. Team 配方（5 角色，与领域无关）
主控(排序、卡闸) / 界定者 / **验收作者(写阶段②的可自检条件)** / 执行者(③④) / 独立评审者(⑤，须不共享被污染上下文)

### 2c. Goal 三件套 schema（结构化，非自由文本）
```
产出物    : <交付什么具体物>
验收断言  : <一条可被 agent/机器自检的成功条件>   # 必须在阶段②先写出
放弃/超时 : <最大迭代数 | 写不出验收断言则中止上报>
```

### 2d. 白送的副产品：**可Loop化闸**
模板的阶段② 逼问一句话：**「这个任务能不能在动手前写出一条可自检的验收断言？」**
- 能 → 它是闭合任务，可 Loop 化。
- 不能 → 它是开放/元层活动，**工厂应拒绝出厂**（而不是造假终止条件 = P0-4 的批量埋雷）。
→ 这把闸**正好解释了 R1 的 P0-2 二分**：修 bug(失败测试)✅、集成构建(契约测试)✅、产品运营(「好的运营」写不出可自检断言)❌。**同一个抽象既给了可复用模板，又给了可Loop化判据**——蓝队认为这是它成立的强证据。

## 3. 迁移测试：把模板套到「集成构建」（另一个闭合任务）
| 阶段 | 集成构建实例化 | 卡不卡？ |
|---|---|---|
| ① 界定 | 「按契约 X 把系统 A 接到系统 B」 | 不卡 |
| ② 建立验收 | 写一条当前会失败的**契约/集成测试** | **不卡**——闭合任务天然能写 |
| ③ 执行 | 写集成代码直到契约测试过 | 不卡 |
| ④ 自检 | 跑契约测试 | 不卡 |
| ⑤ 评审 | 独立查 + 是否碰生产凭证/数据迁移(敏感) | 不卡 |
| ⑥ 交付闸 | 契约测试绿 且 无敏感 → 自动；否则人工 | 不卡 |
→ 模板**看起来**干净迁移到了集成构建。复用的是「6 阶段 + 验收先行」骨架，领域差异只落在阶段②③的具体内容。

## 4. 红队请证伪以下蓝队主张（往死里打）
- **C1**：可复用单元是「闭合任务模式 + 验收先行」，而非一次性领域脚本。
- **C2**：阶段② 的「验收先行」是真正可迁移的承重动作，不是换皮。
- **C3**：「能否事先写出可自检验收」这把闸，正确切分了可/不可 Loop 化。
- **C4**：这套模板让 Builder 比手搓 `/loop` 真省事(预编排 Team + 模板 + 自动卡闸)，值得切换。
