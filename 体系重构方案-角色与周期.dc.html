01

诊断 · 七步偏了，指标钉死了

现在的七步法把一个项目压成一条直线。两个结构性问题：流程偏分析、弱开发；指标被当成步骤，而不是贯穿全程的方法论层。

现在 · 7 步一条直线

开发 · 太薄
运维 · 1
真实 workflow · ECC / Claude Code
需求分析 → 需求设计 → 需求开发 → 运营运维
成熟流程把「开发」展开成设计 + 开发两段；我们的七步却把它压成「原型创建」一格。底层逻辑本就建立在这些开发 workflow 之上 —— 该补回来。
问题 A
偏分析 · 弱开发
7 步里 5 步在分析，开发只剩「原型创建」一格。重心压在「想清楚」，落不到「做出来」。
问题 B
指标当步骤
北极星 / 引领 / 滞后 是贯穿每个阶段的纵向层，却被钉死成第 3 / 4 / 5 步，做完就不管了。
问题 C
角色塌缩
旧开篇把「5 角色 → 1 Builder」，抹掉了真实的角色生命周期。角色不是被 AI 消灭，而是被重新定义。

02 · B　角色配比随周期漂移

同一批角色，不同周期、不同配比。项目重心随产品阶段推移而偏移 —— 这正是「项目要不要走完三个大周期」的答案：它始终是这五种角色，只是配比在变。

02 · D　三个关键洞察
1角色 ≠ 职能
五种原型与岗位头衔无关 —— 设计师可以是 1，也可以是 2 或 3；工程师同理。决定角色的是「擅长做哪类事」，而非「在哪个部门」。
2一人多角色
多数人横跨 2 个角色，少数跨 3 个。角色组合比单一标签更能描述一个人的真实价值。
构建者 + 清理者 = 一个能落地又能收拾的人
3产品阶段决定角色配比
新产品 / 未达 PMF
1 + 2 + 3
造点子 · 能落地 · 能收拾
找到 PMF / 增长期
2 + 3 + 4 +少5
落地 · 打磨 · 增长
PMF 稳固 / 成熟期
3 + 4 + 5 +少2
打磨 · 增长 · 守成
03

落地 · 创建时就声明「这一程侧重哪个周期」

方法论要落到工具上。项目创建时选一个侧重周期，系统据此预置角色配比与侧重阶段 —— 项目不必每次都走完三个周期，而是带着重心出发，再随阶段漂移。

更深一层 · 这套模型读进组织
从「职能分工」到「价值流分工」
传统组织按「写代码 / 画设计 / 做研究」切分；未来或按「谁负责创造、谁负责落地、谁负责打磨、谁负责增长、谁负责守成」切分。当 AI 让跨域行动成本下降，单人可覆盖的链路变长，职能壁垒自然松动。
团队健康度 = 角色配比与阶段的匹配度
不是招「最强的人」，而是招「当前阶段最缺的角色」。全员 1+2、没有 3 的团队会陷入「只造不收」的泥潭；全员 5、没有 1 的团队会失去创新。
角色是动态漂移的
同一个人随产品阶段推移，角色重心会迁移（从 1 → 2 → 4 / 5）。组织需要容忍并支持这种演化 —— 这也是项目从「探索」漂向「运维」的微观写照。
