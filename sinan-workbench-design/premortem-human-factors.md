# 人因 / 落地 蓝军 · Pre-mortem(R3 · 第四轴)

> 作者:编排者代行的人因蓝军(与 RuleGov / 安全 / 成本三轴并行,补一条它们都没碰的轴)。
> 第一性问题:**设计在纸面全对,但"人"这一侧会怎么让它失败?** 这个产品把"判断力"当核心资产,而判断力住在人脑里——人脑会疲劳、会误校准、会退化、会甩锅。这些不是软问题,是会让自治系统在 12 个月内退化回人肉运维或被弃用的硬失效模式。

---

## H1 · 人对 agent 的信任误校准没有被度量 — P0

**第一性原因.** 系统度量了 agent 侧的 false-autonomy(该升级没升级),却没度量**人侧的信任误校准**。人面对 agent 会快速走向两个极端之一:
- **过度信任**(C1 的人侧版本):approve 多了就形成"反正它一般对"的惯性,看 diff 越来越快 → 高危区的橡皮图章。§6 的强制摩擦能挡一部分,但挡不住"看了 diff 但没真看进去"。
- **信任不足**:被坑过一次后,人开始复查一切,把"自主率"硬生生拉回 0,自治 ROI 归零。

两个方向都让"受质量约束的自主率"失真,但**当前没有任何指标能看见人正在误校准**。

**必须解决项.** 引入**人侧校准指标**(与 agent 侧 false-autonomy 对偶):
- **人工否决命中率** = 人 reject/改写的改动里,事后证明"确实该否"的比例。长期偏低 = 过度复查(信任不足);长期偏高且否决率本身低 = 可能过度信任(放过了不该放的)。
- **approve 后回滚率 vs 整体回滚率** 的差:若人批准过的反而回滚更多 → 这个人/这类决策在过度信任。
- 把校准做成**对人的反馈回路**:"你最近 50 次 approve 的下游回滚率高于均值,已为你插入一次强制抽查"。让人能看见自己的校准漂移。

## H2 · 队列永不归零 = 慢性过载,质量随疲劳衰减 — P0

**第一性原因.** §5.4 队列流量把"人是瓶颈"显性化了,这是对的;但设计默认了"净增<0 就健康"。真实风险是:**即便净增≈0,只要队列长期非空且条条高 stakes,人就处在慢性应激**。决策质量是体力活,会随当日疲劳、连续决策数衰减(司法界"午餐前后判决差异"是实证)。一个永远有 5 条高危等着你的队列,本身就是 morale 与质量的慢性杀手——"零卡片成功态"如果一年到不了一次,它就是个嘲讽。

**必须解决项.**
- **决策预算是人的,不只是系统的**:给每个责任人**每日高危决策上限**,超限后系统强制把新高危**重路由**给他人或降派发,而不是堆给同一个人。
- **疲劳感知**:连续决策数 / 时段纳入路由权重;同一人连续 N 条高危后插入强制冷却或转交。
- **把"人均队列驻留时长"和"单人积压峰值"做成一等运营指标**(不只看系统吞吐),并设红线。**这是 §6 瓶颈解法漏掉的人侧。**

## H3 · 技能退化:最难的决策落在越来越不会判的人手上 — P0

**第一性原因.** 这是自治系统最阴险的长期失效。agent 干掉 95%,人只剩最难的 5%。但**判断最难的 5% 所需的手感,恰恰来自做那 95% 的日常练习**——你若一年不写鉴权代码,就越来越没能力审鉴权改动。于是:**随时间推移,最高 stakes 的决策落在持续 deskill 的人手上**,而他们还以为自己能判(过度自信)。航空业的"自动化惊讶"同款:平时脱手,出事时没手感接管。

**必须解决项.**
- **接管/介入面必须做"上下文重建"**,不能只甩一个 diff:给出"为什么走到这、历史同类怎么判的、这块代码的来龙去脉、相关规则"。把人从 deskill 状态快速拉回可判断状态(联动 §7 接管语义)。
- **刻意保留人的"在环练习"**:复审抽样(§5)不只防 gaming,也兼作**保持手感的训练流**——让人定期接触中低危真实案例,而非只在灾难时被叫醒。
- 度量**接管后的人均解决时长趋势**:若持续变长 = deskill 信号。

## H4 · 问责模型缺失:人批了 agent 的活、炸了线上,谁负责? — P1

**第一性原因.** 人在时间压力下 approve 了 agent 的改动,它上线炸了。如果人要为"自己没真写、只是盖章"的代码背全责,理性反应只有两个:**要么过度复查(杀吞吐),要么离职**。当前设计有 provenance(谁批的),但没有**问责语义**——"批准"到底等于"我全责作者"还是"我在 X 信息/Y 时限下做了合理判断"?这个不定义清楚,§6 的强制摩擦会被人理解为"给我留把柄",从而抵制使用。

**必须解决项.**
- 定义**批准 ≠ 全责作者**:provenance 记录"人在多少信息、多长时间、什么 VaR 提示下批的";问责按"是否做了与 VaR 相称的尽职"判定,而非"出事就怪最后点按钮的人"。
- 高危四眼天然分摊问责;低危自动放行的责任归**规则与规则的提升者**(联动 §4.6 Promotion 评审),不归"碰巧在场的人"。
- 让人**信任这个问责模型**是采用的前提——否则再好的 UI 也没人敢点"批准"。

## H5 · 认知负荷与上手门槛:三角色 + 双轴 lens + DAG + 规则,新人怎么活 — P1

**第一性原因.** R1 设计自己也列了风险(双轴 lens 可能过度设计)。叠加 DAG 状态机、五元组、规则账本、VaR——一个新工程师第一天面对的是一个**专家级仪表盘**。若上手要两周,团队会绕开它用老办法,产品名存实亡。"对标 Mavis 的友好"恰恰是在对抗这个,但友好不能只靠拟人动画。

**必须解决项.**
- **渐进式披露 + 角色化默认的极简起点**:新人默认只看到"该你的 N 条决策"这一件事,其余(lens/DAG/规则)按需展开。把 §3 的双轴 lens 默认收成单轴,高级用户再开(采纳 R1 UX 自己的担忧)。
- **每个决策卡自解释**:不需要懂整个系统也能判这一条(why + 风险 + 选项 + 后果),把系统复杂度挡在单次决策之外。

## H6 · "复利账本"会变成又一个没人维护的资产坟场 — P1

**第一性原因.** 刃B 让规则不断产生(approve-all-similar 一次产一条)。一年后是几千条规则。**谁负责让它们保持健康?** 若没人 own 规则集的衰退(世界变了,旧规则开始误放行),复利账本就从"资产"变成"负债坟场"——和它要根治的"Agent.md fork drift"是同一个病,只是搬到了规则层。§4.6 定了精度抽审+auto-demote,但没定**谁对规则集的长期健康负责**。

**必须解决项.**
- 规则**默认带 TTL / 复评周期**:到期未被复评的规则自动降级回 shadow,强制"用进废退"。
- 规则集健康纳入资产 owner 的 JTBD 与度量(规则集腐烂率、僵尸规则占比),让它有主。
- 把"规则也是要被治理的资产"明确写进 §9 配置面治理(规则 = 第六类受治理资产)。

## Top 3 人因 P0(会让产品在 12 个月内退化/被弃用)
1. **H1 信任误校准无度量** —— 加人侧校准指标(人工否决命中率 / approve 后回滚差),与 agent 侧 false-autonomy 对偶。
2. **H2 队列慢性过载** —— 决策预算是"人的":每日高危上限 + 疲劳感知重路由 + 人均驻留/积压峰值红线。
3. **H3 技能退化** —— 接管面做上下文重建 + 复审抽样兼作保持手感的训练流 + 监测接管时长趋势。

## 一句话裁定
**刃A 在"系统能力"上已活,但在"人能否长期用好它"上是未验证的赌注。** 上面 6 条不解决,这台路由器+复利机会在人这一侧慢性失血:要么人疲劳/退化导致质量塌、要么人不信任/绕开导致采用率塌。**人因不是上线后再补的运营问题,是必须进设计的一等约束。**
