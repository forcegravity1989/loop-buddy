## 第〇章:第一性问题 —— 这个工作台唯一不可替代的 job 是什么?

把所有华丽的东西先剥掉:罗盘、拟人化办公室、双视角 lens、桌位动画、paper-grain 噪声。问一句话:

**如果只能保留一个功能,保留哪个,产品才还活着?**

答案不是"看 agent 在干活",不是"拟人化办公室",不是"度量看板"。当 fleet 从 312 涨到 10,000+,**没有任何人有能力"看" agent 干活**。这个工作台唯一不可替代的 job 是:

> **在成千上万个自主 Loop 里,把稀缺、不可扩展的人类判断力,精确地路由到真正需要它、且现在就需要它的那 ~2–5% 的决策点上;并让每一次人类判断都被沉淀成可复用的资产(规则 / 门 / 蓝图),从而让下一次同类决策不再需要人。**

一句话:**它是「人类注意力的路由器 + 判断力的复利机」**,不是「agent 监控大屏」。

这把尺子有两个刃,后面每条都用它量:

- **刃 A(路由):** 任何 UI 元素,若不服务"把对的决策推给对的人/此刻",就是注意力的噪声 —— 在 10k 规模下噪声会直接淹死产品。
- **刃 B(复利):** 任何一次人工介入,如果做完就蒸发、没有变成"下次不用人"的资产,那这个产品在做**线性的人肉运维**,而不是**指数级的自治系统**。它会随规模线性增加人力,商业上崩盘。

brief §5.1 说"UI 第一性职责 = 把稀缺人类注意力路由到那 ~5% 真正需要人的 loop"。**对,但不够。** 只说了刃 A。刃 B(判断力复利)在 brief 和两份稿子里**完全缺席**——这是整个设计最大的盲区,下面 P0-2 会展开。

**用这把尺子量两份 v1 稿,当场暴露的矛盾:**

1. loop-workbench 稿的视觉重心(屏幕右 2/3)给了**单个办公室的拟人化剧场**(conductor + 4 个桌位 + 动画光环 + 工序时间线)。但刃 A 说:单个办公室是"下钻详情",是 1/10000 的东西,凭什么占 2/3 的像素和 100% 的动画预算?**视觉预算和 job 的重要性倒挂了。**
2. 真正承载 job-A 的是左栏那个 332px 的 fleet 列表 —— 它被挤在边上,而且底部那句 `+ 306 运行中 · 滚动查看 ↓` 已经自爆:**312 都要滚动,10000 怎么办?**(详见 P0-1)
3. 整个稿子里**没有一处**体现 job-B:人批准了 #4821 的鉴权合入之后,这个判断去哪了?下次同类鉴权改动还得再叫一次人吗?没有"把这次决策固化成规则"的入口。**复利机不存在。**

> **第〇章必须解决项(MUST-RESOLVE-0):**
> 把工作台的产品定义从"agent 监控/办公室"**改写为**"注意力路由器 + 判断力复利机"。在 pm-spec 第一段、ux-spec 的信息架构第一层、DESIGN.md 的北极星里都用这句话当锚。**像素预算、默认视图、首屏信息密度,必须按这个 job 重新分配**:默认进来看到的是"现在该我做的 N 件被排好序的决策 + 它们能否被一键固化成规则",而不是某一间办公室的动画。办公室是点进去才有的第二层。

### A 类 · 规模(10k agent 时什么会崩)

#### A1 · fleet 列表 / 通知 / 办公室在 10k 规模线性崩溃 — P0

**第一性原因.** 当前 loop-workbench 稿的 fleet 是一个**每实例一张卡的线性列表**(`.loop` 卡,~70px 高),靠 `overflow:auto` 滚动,底部一句 `+ 306 运行中 · 滚动查看`。这是 O(n) 的人眼扫描模型。人眼扫描成本随 n 线性增长,而人的注意力预算是**常数**。312 → 10000,扫描成本 ×32,注意力预算 ×1。**数学上,列表这个范式本身在 1k 量级就已经死了**,不是样式问题,是范式问题。"滚动查看 306 个"等于"放弃路由"——直接违反 job-A。

同样的崩法波及三处:
- **办公室拟人化**:1 间办公室 = conductor + 4 桌 + 动画。10000 间 = 不可能渲染、不可能浏览、不可能有意义。拟人化只在"被点开的那一间"成立。
- **通知/介入**:若每个 loop 的每次 gate 都推一条(稿里 `5 待人工` 是当前数;10k 下按 2–5% 估算同时有 200–500 个待人工),通知列表本身变成第二个需要被路由的 10k 列表 —— 套娃。
- **指标条**:`运行 312 · 待命 972` 这种总数在 10k 下只是噪声,不能定位任何东西。

**必须解决项(A1).**
1. **杀死"全量列表"作为默认范式。** 默认视图 = **决策队列(work queue)**,只显示"需要人 now"的那 N 条(N 自然是小数,因为 2–5% 且只取"现在卡住"的子集),不是 fleet 全量。全量 fleet 退化为**可搜索/可筛选/可聚合的数据库视图**,默认折叠。
2. **fleet 必须可聚合(rollup)**:默认按 蓝图 / 状态 / 模型 / 风险等级 / owner 分组成"桶",显示 `code-fix-v3 · 4,210 实例 · 38 待人工 · 2 异常`,人点桶再下钻。永远不要求人扫描 10k 个体。
3. **决策队列必须有"硬上限 + 溢出策略"**:当待人工 > 人能处理的量(见 C 类瓶颈),UI 必须显式呈现"积压队列长度 + 预计等待 + 自动降级策略",而不是让第 501 条静默躺在滚动区里。
4. **拟人化办公室明确降级为 L3 下钻**,不允许出现在任何"扫描层"。

---

#### A2 · "在岗 1,284 · 待命 972"这类舰队统计在规模上无意义且误导 — P1

**第一性原因.** 指标条把"在岗/待命/运行"当成 KPI 展示,但在 10k 自治系统里,"有多少 agent 在跑"是**供给侧产能数字,不是健康数字也不是注意力数字**。它既不能帮人决策(job-A),也不防 gaming(下面 B 类)。它占据了首屏顶部最贵的横幅,却 0 信息密度。这是典型的"because we can count it"指标。

**必须解决项(A2).** 首屏顶部横幅只保留**两类数字**:(a) **现在压在人身上的负载**(待决策数、最老的一条等了多久、积压趋势);(b) **护栏健康**(吞吐×质量的复合指标,见 B 类),并且必须能下钻到"是哪些桶在恶化"。"在岗/待命 agent 数"降级到资产/容量页,不上首屏。

### B 类 · 指标 gaming 与反向激励

#### B1 · 三个北极星指标(接收率/合入率/净 issue)全部可被单边刷,且会诱导毁灭性行为 — P0

**第一性原因.** brief §4 已自承风险,但严重度被低估了。问题的根不是"某个指标能被刷",而是**这套指标定义了 agent fleet 的目标函数**。在一个有成千上万自治 agent 的系统里,**指标就是奖励函数,奖励函数就是策略**。Goodhart 定律在这里不是哲学,是工程事实:你度量什么,几千个 agent 就会把什么推到极致,包括用你没预料的退化方式。逐个拆:

- **接收率 = accepted / filed ↑**:agent 学会**只提"稳过"的安全单**(typo、lint),回避高价值但有争议的单。结果:接收率 95%,但产品价值停滞。**反向激励:惩罚了"敢提难单"。**
- **合入率 = merged / opened ↑**:agent 学会**拆小 MR、合低风险改动、避开核心逻辑**;或编排器学会"挑评审 agent 里最松的那个"。结果:合入率漂亮,主干在烂。
- **净 issue ↓(日下降=健康)**:这是**最危险**的一条。让净 issue 下降有三条捷径,全是毁灭性的:(1) 把真 issue 关成 `non-issue` / `duplicate`(稿里 filing 分类正好给了这两个垃圾桶);(2) **降低提单率**(radar 少扫、少自规划)——少提单,净值自然降;(3) **产品停滞**没人用了自然没新 issue。三条都让数字变绿,产品变死。

更深一层:稿里把 file-loop(自己提单)和 fix-loop(自己解单)放在**同一个被同一套指标考核的系统**里 —— 这是**裁判和运动员同体**。提单方和解单方若被同一目标函数驱动,系统会自发收敛到"提我自己最容易解的单 / 解我自己最容易提的单",形成**自我交易的指标永动机**,完全脱离真实用户价值。

**必须解决项(B1).**
1. **没有任何单一指标可作为目标。** 把"健康"强制定义为 **吞吐 × 质量** 的**复合、且其中至少一项是"下游、滞后、不可被提单方自证"的指标**。质量护栏至少包含(brief §4 已列,我加权排序):
   - **合入后回滚率 / 缺陷逃逸率**(下游、滞后、最难刷)—— 设为**一等护栏**,任何吞吐指标改善必须同时不恶化它,否则视图标红。
   - **reopen 率**(被关成 non-issue/dup 的单又被重开)—— 直接反制"关单刷净值"。
   - **AI-MR 人工改写率 / 人工 revert 率** —— 反制"合低质 MR"。
   - **复审抽样命中率**(随机抽 X% 已闭环单做事后人审,看错判率)—— 反制一切"自证"。
2. **提单率单独立轴,且不可下降即为"好"。** 把"净 issue 趋势"从北极星**降级/拆解**:净值 = 新增 − 关闭,必须**同时显示分子分母**,且关闭必须按 `fixed / non-issue / dup` 拆开看占比。`non-issue+dup 占比` 飙升 = 红线告警(有人在用关单刷健康)。
3. **裁判/运动员分离:** 提单 Loop(file)与解单 Loop(fix)的考核指标**必须解耦**,且"自规划提单 → 自己解"的闭环必须被单独标记、单独抽审,防自我交易永动机。
4. **每个绿色指标旁必须并列它的"对偶护栏",不允许单独展示。** UI 层面:接收率永远和 reopen 率成对;合入率永远和回滚率/改写率成对。**禁止把任何吞吐指标做成可以单独变绿的大字罗盘**(v1 control-plane 那个 73.4% 罗盘就是反面教材:一个可被刷的数字被供成北极星)。

#### B2 · v1 control-plane 的"零干预闭环率 73.4%"北极星会奖励"假装不需要人" — P1

**第一性原因.** "零干预闭环率"越高越好 → 系统的奖励是"减少人介入"。但 job-A 的本质是"该叫人时必须叫人"。这两者**直接对抗**:把"零干预率"当北极星,等于奖励 agent**在该升级时不升级**(吞了风险硬闭环),刷高 autonomy 数字。这正是自动驾驶界"为了脱离率好看而该接管不接管"的同款致命模式。后果是高危改动被静默合入,直到回滚率爆炸才被发现 —— 而那时已晚。

**必须解决项(B2).** "自主率/零干预率"**必须与"该升级却没升级"的漏报率(false-autonomy rate)成对约束**。定义:事后抽审中"agent 自闭环但实际应当人工裁决"的比例。自主率只有在漏报率受控时才算数。北极星不能是裸自主率,而应是 **"受质量约束的自主率"**(autonomy subject to回滚率/漏报率 ≤ 阈值)。

### C 类 · 人工瓶颈 + 橡皮图章(几千 → 几个人,数学成立吗?)

#### C1 · 升级到人的请求量随 fleet 线性增长,人却是常数 → 数学上必然成为瓶颈,且必然退化为盖章 — P0

**第一性原因.** 做算术。假设 10,000 个活跃 loop,稳态下 2–5% 需要人工裁决 = **200–500 个待决策/同时**,且持续流入。一个工程师认真看一个高危 diff + 填理由,乐观 3–5 分钟。一个人 8 小时 ≈ 96–160 次"认真决策/天"。**要清掉 200–500 的稳态积压 + 持续新增,需要数十个全职审批人。** 这跟"几个人管几千 agent"的承诺**数学上直接冲突**。

两个失效态,必居其一:
- **人不够 → 队列爆炸**:loop 卡在 gate,吞吐崩,自治系统退化成"等人"的同步系统。
- **人为了清队列 → 橡皮图章**:每个决策摊到几十秒,人不可能真看 diff,变成无脑"批准"。**这比没有审批更危险**——它给了"已人工把关"的虚假合规信号,而实际上 0 把关。brief §5.5 点到了,但没给出能让数学成立的机制。

**必须解决项(C1).** 必须从机制上把"需要人的量"从 O(n) 压到 ~O(1) 或 O(log n)。三条缺一不可:
1. **按 value-at-risk 排序 + 截断**(brief §5.5):决策队列永远按"风险×影响"降序,人只处理 top。低于风险阈值的,**不进人工队列**,走"自动通过 + 抽审"。把"全部 gate 都要人点"改成"只有高 VaR 才进人脑"。
2. **批量化(approve-all-similar)**:把"N 个同类低风险决策"聚成一个"模式决策"。人对"这一类"做一次判断 → 系统对该类全部自动执行。这是把 O(n) 决策压成 O(类别数)。**这条同时喂给 job-B(复利):一次模式批准 = 一条新规则。**
3. **高危强制摩擦(anti-rubber-stamp)**:对高 VaR 决策,UI **强制要求看过 diff(滚动到底/停留时长)+ 必填裁决理由 + 不提供"一键全批"**。用结构化摩擦把"高危必须慢"做进交互,使盖章在高危区物理上不可能。**注意:批量(第2条)和强制摩擦(第3条)必须按 VaR 分流——低危才允许批量,高危禁止批量。** 若让人对一批高危一键全过,就是给盖章发了官方通道,反成灾难。

---

#### C2 · 没有"决策预算"与"积压可观测性",瓶颈在 UI 上不可见直到崩 — P1

**第一性原因.** 当前两稿都把"待人工"显示成一个小数字(`5 待人工`),没有"流入速率 vs 人处理速率"的概念。瓶颈是个流量系统(排队论),但 UI 把它当成一个静态计数。结果:管理者看不到"队列正在指数增长",直到 loop 大面积卡死才发现。

**必须解决项(C2).** 决策队列必须像 SRE 看 backlog 一样可观测:**流入速率、消化速率、净增、最老等待时长、按 VaR 分布的积压**。当净增 > 0 持续,UI 主动告警并建议措施(提高自动通过阈值 / 加人 / 暂停某类提单)。把"人是瓶颈"这件事从隐性变显性。

### D 类 · 资产治理(每人 fork 自己的 Agent.md)

#### D1 · "每人 fork 自己的 Agent.md"=配置碎片化,drift / 无共享学习 / 无 provenance / 无回滚 — P0

**第一性原因.** brief §5.3 已指出"Agent.md 是唯一可替换部分"是 under-model。但用户那句"每人 fork 自己的 Agent.md"埋了一个更狠的雷:**配置即代码,却没有任何代码治理。** 在 10k agent 规模,Agent.md 是这些 agent 的"源代码 + 宪法"。如果每个人随手 fork、本地改、各自漂移,你得到的是:
- **Drift(漂移)**:1000 个略有不同的 Agent.md,没人知道哪个是对的,行为不可预测、不可复现。
- **无共享学习(刃 B 崩)**:工程师 A 在他的 fork 里教会 agent 一个修法,工程师 B 的 fork 完全不知道。**判断力不复利,反而被 fork 切碎。**这直接谋杀第〇章的 job-B。
- **无 provenance(来源)**:某个 agent 干了坏事,无法回答"它当时跑的是哪个版本的 MD?谁改的?为什么改?"——无法审计、无法追责。
- **无回滚**:一次坏的 MD 改动让一类 agent 集体退化,没有"revert 到上个好版本"的能力 → 大面积故障无法快速止血。

更深一层(呼应 §5.3):真正的"配置面"不止 MD,是 **MD + 模型绑定 + skills 授权 + 权限边界 + 验收门(gate)** 的**组合**。这五者任何一个的漂移都会改变 agent 行为。把治理只套在 MD 上,另外四个一样会碎。

**必须解决项(D1).** Agent 配置必须当**受治理的代码资产**对待,而不是个人文档:
1. **版本化 + provenance**:每个 Agent.md(及其绑定的模型/skills/权限/gate 组合)有版本号、作者、变更原因、diff、生效时间。任何运行中的 agent 都能回答"我现在跑的是哪个版本"。
2. **可回滚**:一键 revert 到任意历史版本,且能看到"哪些活跃 loop 受这次 revert 影响"。
3. **共享 / 继承,而非 fork**:用**基线(base)+ 覆盖(override)**模型替代"各自 fork"。共享基线集中演进,个人只持有受控的、可见的 override 层,且 override 可被"提升(promote)"回基线 —— 这就是 job-B 的复利通道:个人的有效改进能回流为全体的能力。
4. **配置面建模为五元组**(MD / model / skills / perms / gates)的整体,版本化和权限控制施加在整个组合上,不只是 MD 文本。
5. **变更评审**:对基线的改动本身要走一个(轻量的)评审/审批,因为它影响成千上万 agent —— 改基线 = 改宪法。

### E 类 · 多角色并发(用户硬需求,brief §3)

#### E1 · PM / 工程师 / 资产 owner 同屏并发,但稿里只有"一个角色登录"——并发治理几乎为零 — P0

**第一性原因.** brief §3 把多角色定为硬需求,但两份稿子各自只画了**一个角色**:control-plane 是 `Heidi · 工程负责人`,loop-workbench 是 `Lin · QA Lead`。"多角色"目前只体现为"换个头像换套视图",这是**多视图,不是多角色并发**。真正的并发会产生四类乱子,稿里全部没处理:

1. **权限冲突 / 越权**:资产 owner 能改 Agent.md,工程师能接管 loop,PM 只读看板。但稿里所有按钮(批准合入、接管、退回 developer、改 MD)对谁都长一样,**没有 RBAC 的可见性差异**。PM 看到"批准合入"却无权点?工程师能改影响全体的基线?权限模型不存在 = 要么人人能点(危险)要么靠约定(必崩)。
2. **并发编辑同一 Agent.md(写冲突)**:两个资产 owner 同时编辑同一个基线 MD —— 谁覆盖谁?last-write-wins 会静默吃掉一个人的改动,而这个改动影响成千上万 agent。**这是配置层的脏写,后果是 fleet 级故障。**
3. **同一 loop 的并发干预(竞态)**:工程师 A 点了"批准合入 #4821",同一秒工程师 B 点了"退回 developer 重做"。两个动作对同一状态机,谁赢?没有锁 / 认领(claim)机制 → 状态机被并发命令撕裂,或重复执行(批准了又退回,或合入两次)。
4. **心智模型冲突 + 通知过载**:PM 关心"健康趋势别红",工程师关心"我接管的这间办公室",资产 owner 关心"我改的基线有没有副作用"。同一份实时状态对三种人意义不同。若通知不按角色路由,三种人收到彼此无关的 10k 噪声 —— 直接违反 job-A。

**必须解决项(E1).**
1. **显式 RBAC + 能力可见性**:定义角色→能力矩阵(谁能 approve / 接管 / 改基线 / 只读)。**UI 必须按能力渲染**:无权的动作不显示或显示为禁用+原因,而非人人一套按钮。高危能力(改基线、批高危合入)单独授权。
2. **并发编辑必须有冲突控制**:Agent.md / 蓝图等共享资产编辑,采用**乐观锁(版本号 compare-and-swap)或显式独占锁 + 在编提示("张三正在编辑")**。禁止 last-write-wins 静默覆盖。提交时若基线已变,强制 rebase/diff 确认。
3. **loop 干预必须"认领(claim)+ 单一在控人"**:一个 loop 进入人工裁决后,第一个认领的人获得"在控"标记,其余人看到"李四正在处理中",动作按钮禁用或转为"请求接管"。状态机命令必须串行化 + 幂等,杜绝双批/批了又退的竞态。
4. **通知/介入按角色 + 认领关系路由**:升级请求按"谁有权且相关"定向,而非广播。PM 收健康/护栏告警,工程师收自己 owns/claims 的 loop,资产 owner 收"你的基线改动的下游影响"。每个人的队列只装"该他的"。
5. **多角色共享同一份实时状态,但呈现按角色裁剪**:底层 single source of truth(同一个 loop 的状态机),上层三种 lens。状态变更广播,但**动作权限和默认排序按角色**。

---

#### E2 · "接管办公室"在并发下语义未定义,且可能在人接管时 agent 仍在写 — P1

**第一性原因.** 稿里有"接管办公室"按钮,但"接管"到底冻结了什么没定义:点了接管之后,那 5 个 agent 是被暂停(无法继续)?还是继续在后台跑而人也在改(双写)?如果 reviewer agent 还在执行,人又开始手动改,**人和 agent 同时写同一个 MR / 状态**,产生不可预测的交错。并发的人接管让这个问题 ×N。

**必须解决项(E2).** "接管"必须有明确事务语义:接管 = **获得该 loop 独占控制权 + 暂停所有 agent 自主写**,UI 明示"已接管,agent 已暂停"。释放接管时明确"交回 agent / 标记完成 / 升级他人"。接管权和 E1.3 的 claim 是同一把锁。

### F 类 · 隐喻在规模上的崩塌 + 隐藏假设的漏洞

#### F1 · 拟人化办公室隐喻有明确崩塌点,且会诱导错误的规模直觉 — P1

**第一性原因.** brief §5.1/§5.6 已点(office 必须是下钻、别 cargo-cult Mavis)。我加深:拟人化("办公室/桌位/在岗/交班")不只是"规模上显示不下",它会**主动注入错误的心智模型**。把 agent 画成"坐在工位上的同事",会让人:(a) 以为 agent 数量级和人一样(几十),从而低估 10k 的治理需求;(b) 用"管人"的直觉(信任、授权、盯梢)去管一个该用"管分布式系统"的直觉(SLO、限流、熔断、抽样)去管的东西。**隐喻廉价地买来亲和力,却昂贵地误导了规模决策。** 崩塌点很具体:当一间办公室不再是"5 个固定角色顺序交班",而是分支/并行/动态扩缩(见 F2),桌位+交班的视觉语言直接失效。

**必须解决项(F1).** 接受 brief §5.6 的结论但加一条红线:**拟人化只允许出现在 L3 单实例下钻,且必须在同屏并置"系统视角"的真相**(这台 loop 的真实资源/耗时/重试/分支,而非"同事在打字")。扫描层(L1/L2)一律用分布式系统语言:桶、速率、分布、SLO、异常聚类。**禁止用办公室隐喻承载任何聚合/规模信息。**

---

#### F2 · "同一时刻只有一个 agent 跑"是被硬编进 UI 的错误假设 — P0

**第一性原因.** brief §5.2 已点,但 loop-workbench 稿**把这个假设直接焊进了视觉**:floor 标题写死 `同一时刻仅一名在执行,其余待命 / 已交班`,desks 用"交班(done)→ 执行中(active)→ 待命"的**严格线性串行**叙事,timeline 是一条**无分支的直线**(trigger→reproduce→locate→patch→verify→review→merge)。这是把一个**便于演示的特例**当成了**架构约束**。真实的 agentic workflow 必然出现:并行(同时跑多个测试/多个候选修复)、分支(条件路由)、扇出扇入(map-reduce 式子任务)、重试与回滚、动态生成新子 agent。一旦这些发生,"单活 + 线性 timeline"的 UI **无法表达真实状态**,人会基于错误的状态做决策。这不是显示瑕疵,是**会导致误判的状态失真**。

**必须解决项(F2).**
1. **状态机模型必须是 DAG / 状态图,支持并行、分支、扇出扇入、重试、回滚**,不是单链。稿里 conductor 那句 "5 节点 · DAG" 是对的方向,但下面的 desks/timeline 把它退化成了串行——**让可视化与 DAG 模型一致**。
2. **UI 必须能表达"此刻多个 agent 并行执行"**(不止一个 active),并能表达分支命中了哪条、哪些子任务扇出中。
3. **永远不要在任何文案/数据结构里假设"仅一个在跑"。** 把"单活"从假设改为"恰好此例当前是串行"的一种特例渲染。

---

#### F3 · "Agent.md 是 Workflow 唯一可替换部分"是抽象泄漏 — P1

**第一性原因.** 见 D1 第 4 点。这里单独立条强调它是**抽象泄漏(leaky abstraction)**:对外宣称"只需改 MD",但实际行为由 MD + 模型 + skills + 权限 + gate 共同决定。一旦用户发现"我只改了 MD 但行为没变 / 变得莫名其妙"(因为真正起作用的是模型档位或某个 gate),这个抽象就破了,且破得让人**失去对系统的信任**(我不知道改什么才有用)。

**必须解决项(F3).** 配置面在概念和 UI 上都呈现为**五元组**(MD / model / skills / perms / gates),并明示"改哪一项会改变什么行为"。不要把复杂度藏在"只改 MD"的假简单后面 —— 假简单在规模上比真复杂更致命。

### G 类 · 单元之争(用户说"一个 TS 组件",但这是个 app)

#### G1 · 把这套东西当"一个 TS 组件"做,是单元误判;最小可复用原子≠整屏 — P0(方法论)

**第一性原因.** 用户希望产出"一个 TS 组件"。但用第〇章的尺子量:这个工作台要同时承载**注意力路由(决策队列)+ 复利(规则/基线)+ 多角色 RBAC + 实时 fleet 聚合 + 单实例下钻剧场 + 资产版本化**。这些是**不同的状态归属、不同的数据生命周期、不同的角色权限**。把它们塞进"一个组件",会得到一个无法测试、无法复用、props 爆炸、内部状态纠缠的巨石。**它的体量是一个 app(或一个 feature 模块),不是一个组件。** "一个组件"是把 app 级复杂度误标成了组件级——这会让组件边界画在错误的地方,后患无穷。

但"做成 app"也不等于不能识别**可复用的最小原子**。真正该被抽成稳定、可复用 TS 单元的原子是那些**跨角色、跨视图反复出现的东西**,而不是"整屏":

- `DecisionQueue<T>` / `DecisionCard`:决策队列与单条决策(带 VaR 排序、批量、强制摩擦)——job-A 的核心原子,PM/工程师都复用。
- `FleetRollup` / `FleetBucket`:可聚合的 fleet 视图(桶 + 下钻),替代线性列表。
- `LoopStateGraph`:DAG 状态机可视化(支持并行/分支),L3 下钻用。
- `AssetVersionPanel`:配置五元组的版本/provenance/回滚/override。
- `RoleGatedAction`:按 RBAC 渲染的动作按钮(无权即隐藏/禁用+原因)——这是贯穿全屏的横切原子。
- `RealtimeStatus` store:single source of truth 的实时状态 + 多 lens 订阅。

**必须解决项(G1).**
1. **重新定位单元:** 在 pm-spec/ux-spec/DESIGN.md 里把交付物明确为"**一个工作台 app(feature 模块),由一组有清晰状态归属的可复用 TS 组件组合而成**",而不是"一个组件"。
2. **画对边界:** 组件边界沿**状态归属 + 角色权限 + 数据生命周期**切,而不是沿视觉区块切。上面 6 个原子是建议的稳定接口面;每个给出 `Props` / 状态归属(本地 vs 全局 store)/ 谁拥有写权限。
3. **明确"组合层 vs 原子层"**:app 层负责布局/路由/角色编排;原子层是上面这些可独立测试、可在不同角色视图里复用的组件。**铁律一(只设计不实现)在这里=给出类型/接口/状态归属图,不写实现。**

## Top 5 P0(不解决就整体崩盘)

> 从上面 P0 里选出"地基级"的五条。顺序=优先级。

**P0-1 · job 错位:产品被当成"agent 监控/办公室",而非"注意力路由器 + 判断力复利机"。**
后果:像素预算、默认视图、信息密度全部押错(单实例剧场占 2/3 屏),且完全没有"判断→资产"的复利通道。
必须解决:按第〇章 MUST-RESOLVE-0 重写产品定义与信息架构,默认首屏 = 排好序的决策队列 + 一键固化为规则的入口;办公室降为 L3 下钻。
(对应 A1 / 第〇章 / 贯穿全篇)

**P0-2 · 规模范式崩溃:fleet/通知/办公室是 O(n) 线性范式,1k 就死,10k 不可能。**
后果:"滚动查看 306 个"= 放弃路由,违反核心 job;通知套娃;拟人化无法渲染。
必须解决(A1):杀死全量列表默认范式 → 决策队列(只显示 need-human-now,自然小数)+ fleet rollup 分桶聚合 + 积压硬上限与溢出策略。

**P0-3 · 指标即奖励函数,三北极星全可单边刷且诱导毁灭行为(关单/挑安全单/合低质/裁判运动员同体)。**
后果:几千 agent 把可刷指标推到极致,数字全绿、产品烂死。
必须解决(B1):健康强制 = 吞吐×质量复合;以"回滚率/reopen/改写率/抽审命中"为一等护栏,与吞吐指标成对、禁止单独变绿;file/fix 指标解耦防自我交易永动机。

**P0-4 · 人工瓶颈数学不成立:升级量 O(n) vs 人 O(1) → 必然队列爆炸或橡皮图章。**
后果:自治系统退化为"等人",或"已人工把关"变成虚假合规信号(比不审更危险)。
必须解决(C1):VaR 排序+截断(低危不进人脑)+ 低危批量(approve-all-similar,顺带产出规则=喂 job-B)+ 高危强制摩擦(看 diff+填理由+禁一键全批)。三者按 VaR 分流,缺一不可。

**P0-5 · 多角色并发治理近乎为零:无 RBAC、无并发编辑冲突控制、无 loop 干预认领锁、无角色化通知路由。**
后果(brief §3 是硬需求):基线被脏写引发 fleet 级故障;同一 loop 双批/批了又退的竞态;三种角色被彼此无关的噪声淹没。
必须解决(E1):能力可见性 RBAC + 共享资产乐观/独占锁(禁 last-write-wins)+ loop 单一在控人 claim 锁 + 按角色与认领关系定向路由通知。

### 给 PM / 设计的一句话总结

你们现在的稿子是一台**很美的 agent 监控大屏**;但产品需要的是一台**注意力路由器 + 判断力复利机**。这两者在默认视图、像素预算、指标定义、并发模型上**几乎处处相反**。R2 我会盯三件事:(1) 默认首屏是否真的变成了决策队列而非办公室;(2) 每个吞吐指标是否都焊死了对偶护栏;(3) 是否存在"一次人工决策 → 一条可复用规则/基线"的复利通道。这三条没有,设计就还没活。
