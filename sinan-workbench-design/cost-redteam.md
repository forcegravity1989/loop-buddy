# 司南 · Loop 工作台 — 经济 / 可行性蓝军终审(cost-redteam.md)

> **攻击轴**:成本与单位经济学。前几轮(规模/指标/瓶颈/并发/复利/安全)碰的都是"能不能跑、跑得对不对、刷不刷得动",**没有一轮问过"跑得起跑不起"**。
> **核心问题**:10k 活跃 agent 规模下,这套系统在财务上活得下去吗?
> **判据**:一个又自主、又高质、但单位闭环成本失控的系统,在商业上和一个低自主率系统**一样会死**——只是死法从"人力线性崩盘"换成"算力线性崩盘"。DESIGN.md 把刃B(复利)定位成"防人力 O(n) 崩盘"的唯一通道,**但全文 13 节、北极星双因子、§5 看板矩阵里没有一个成本指标**。这是一个系统性盲区,不是某条遗漏。
> **方法论前提声明**:下列费米估算的绝对数取决于"每 agent 每天调用次数 × 每次 token 量 × 单价",这三个量设计文档**均未给出**。我用业界 agentic 工作流的合理档位(见 §0)给出**量级**和**敏感度**,重点不是"精确多少美元",而是"哪些设计选择把成本推向哪个数量级,以及谁在踩刹车"。

## 1. 推理成本:10k agent 的日/月 token 成本量级 + "Opus 泛滥"失控风险

**场景**
10k agent 全天候跑 file/fix 两类 loop,加上复利机的 shadow 运行(§2)、复审抽样重判(§5)。运营三个月后没人盯过账单,直到财务发现单月推理支出超过整个团队人力成本。

**费米估算(中档,6B tok/日)**
按 Opus:Medium = 30%:70% 的 token 切分,**blended 单价 ≈ 0.3×$30 + 0.7×$2 ≈ $10.4 / 1M tok**:

- **日成本** ≈ `6,000 M tok × $10.4/M ≈ $62k/日`
- **月成本** ≈ **$1.8M–$1.9M/月**,**年 ≈ $22M**。

**这一步最反直觉的事实——成本由"混合比"主导,不由"总量"主导。**
同样 6B tok/日,只把 Opus 占比从 30% 推到 70%(token 维度),blended 单价跳到 ≈ $20.4/M,**月成本翻到 ≈ $3.7M**。而 Opus 单价本身比 Medium 高约 15×,所以:

> **30% 的 Opus 调用,吃掉约 85% 的账单。** 成本曲线对"Opus 占比"这一个旋钮**极度敏感**,对"总调用数"反而是线性温和的。

**敏感度三联(说明量级带宽)**

| 情景 | 日 token | Opus 占比 | 月成本量级 |
|---|---|---|---|
| 节制(低档调用 + 10% Opus) | 0.5B | 10% | ~$70k–$100k/月 |
| 中档(主用) | 6B | 30% | ~$1.8M/月 |
| **Opus 泛滥(高档)** | 18B | 70% | **~$11M/月** |

最坏与最好之间是 **~100× 带宽**。设计文档没有任何机制把系统钉在带宽的低端。

**严重度:P0**
绝对量级(月 $1.8M / 失控 $11M)足以单独决定项目生死;且失控方向(Opus 泛滥)正是当前设计**没有任何护栏**的方向。

**第一性原因**
DESIGN §9 用**自然语言**定义绑定规则:「复杂→Opus,简单→Medium/Haiku」。"复杂"无操作化定义、无量化阈值、无配额。在一个**奖励"自主率×质量"而完全不惩罚成本**的系统里(§6 看板无成本维度),任何理性的 agent/owner/规则都会发现:**升到 Opus 几乎总能让质量护栏更容易过、让自主率更高**——成本的负反馈不存在,于是 Opus 占比单调上漂。这就是 §3 要展开的治理空洞在推理层的直接后果。

**必须解决项 + 建议成本护栏指标**
1. **`blended_cost_per_1M_tok`(混合单价)= 北极星级别的一等成本护栏**,看板常驻。它一个数同时反映"调用量"和"Opus 占比",是最高杠杆的单一指标。
2. **`opus_token_share`(Opus token 占比)单独成线 + 红阈**(如 >40% 触发审查)。因为账单 85% 由它决定,它必须像 §5.2 护栏一样"破阈一票否决"。
3. **`cost_per_closed_loop`(单位闭环成本,见 §6)= 与"自主率×质量"并列的第三北极星因子。**
4. **硬性月度 token 预算 + 燃尽告警**:无总预算上限,任何按调用计费的 agentic 系统都会无声膨胀。

## 2. 复利机的成本悖论:省人力 vs 烧算力,净账到底正不正?

**场景**
复利机按 §4.1 强制 shadow-first:每条新规则先**影子运行**——"它预测决策,人仍然亲自决定"。于是对每一个进入 shadow 的 escalation,系统**既跑了规则推理、又付了人工成本**(双倍,且人力一分没省)。同时 §4.3/§5.2 要求 active 规则自动解决的案例**仍按比例进复审抽样池**做事后人审。运营半年后:人力曲线如期下降,**但推理账单不降反升**——因为复利机本身是个吃算力的机器。

**这是本次审查最锋利的发现,值得拆成三笔账。**

**账(a)—— Shadow 期是纯负投资,且烧的是"双份"。**
DESIGN §13 R-7 已诚实承认"shadow 期人仍要判,见效前是负投资"——**但它只算了人力的负投资,漏算了算力的双倍支出。** Shadow 阶段:
- 人力成本:**100% 照付**(人仍亲自决定);
- 算力成本:**额外 +1 份**(规则的预测推理);
- 净效果:**该时段单位决策成本 = 人力 + 额外算力 > 不上复利机**。

复利机在转换期**同时**烧更多算力**和**不省人力。R-7 把它标 P1 是**低估**了——因为它没看见算力那一份。

**账(b)—— 稳态:省下的是"廉价人秒",烧掉的可能是"昂贵 Opus 秒"。**
规则毕业后自动解决一类 escalation,省下人工决策。但要问:**省的和烧的,单位价值对不对得上?**
- 省下的:一次人工决策,边际成本 ≈ 人秒(见 §4 估,单次 $1.7–$8 量级)。
- 烧掉的:① 规则自动解决时的**推理调用**(若该类被判"复杂"绑了 Opus,单次可能 $0.3–$3+);② **永久性的复审抽样人审**(active 规则不是一劳永逸,§5.2 要求持续抽审,这份人力**永不归零**);③ shadow→canary→active 全生命周期的精度抽审与下游 reopen 信号采集(§4.6③)开销。
- **净账隐患**:对"低 VaR、本来就该走自动通过+抽审"的长尾类别,人工决策本来就便宜(甚至接近零,因为 §6 截断规则让它"不进人脑"),复利机为它建规则、跑 shadow、做抽审,**很可能省下的人力 < 烧掉的算力+残留人审**。

> **结论:会出现"复利在省钱的同时更烧钱"。** 准确说是——**复利在"省人力曲线"上向下,在"总成本曲线"上可能向上**,二者被设计文档当成同一条线(R-7 只画了人力线)。这是 §6 盲区的直接后果:没有总成本视图,就看不见这把剪刀差。

**账(c)—— Shadow 成本随"流量 × 候选规则数"二次增长。**
关键放大器:**一个 escalation 流过时,所有正在 shadow 的相关规则都要对它跑一次预测推理。** 若同时有 K 条规则在 shadow,则该类流量的影子算力 ≈ K×。而 §4.6④ 的死锁修复("保留 shadow 样本下限、优先喂临近毕业规则的流量")为了**喂饱样本**,会**主动维持 shadow 流量**——即主动维持这份双倍/K 倍算力。**为防样本饥饿而保流量 = 为防慢而花钱,这笔钱没有任何指标在盯。**

**严重度:P0**
复利机是 DESIGN 全文的"明星机制"和唯一的 O(n)→次线性通道(§4.1、§13 R-6)。若它的真实净账在相当一部分类别上是"更烧钱",则**整个商业可行性论证的地基松动**:你以为买的是"省钱机器",实际可能是"用算力换人力、且汇率不划算"的机器。终审 blueteam-r2 已实证"复利机最少被自我质疑"——成本维度正是它又一个没被质疑的盲区。

**第一性原因**
复利机的设计**只优化"人工决策数"这一个量**(决策杠杆、复利曲线、savedHumanDecisions 全是"人次"单位),从未把"算力成本"放进它自己的损益表。它度量的是"省了几次人",不是"省了多少钱"。在人秒便宜、Opus 秒昂贵的成本结构里,"省人次"和"省钱"不是同一件事。

**必须解决项 + 建议成本护栏指标**
1. **`net_dollar_savings_per_rule`(每条规则的净美元节省)= savedHumanDecisions × 人工单次成本 − (规则推理成本 + 残留抽审人审成本 + 摊薄的 shadow/canary 培育成本)。** 这是复利机损益表里**唯一诚实的指标**;必须**带置信下界**(沿用 §4.6③ 对反事实估计的要求)。
2. **`rule_payback_period`(规则回本周期)**:从 shadow 启动到累计净美元节省转正的时间。**净节省永远转不正的规则 = 反向复利,应 auto-demote / 不予 promote。** 这直接补上 R-7 缺的"何时回正"判据,且用**美元**而非"决策杠杆>1"来判(后者只管人次,会漏掉"省了人但更烧钱"的规则)。
3. **`shadow_compute_overhead_ratio`(影子算力开销比)= shadow+抽审总算力 / 该域基线算力**,设上限。防止 §4.6④ 为喂样本无限维持双倍/K 倍算力。
4. **复利曲线必须出双线**:现有"每千 loop 人工决策数"(人力线)**叠加**"每千 loop 总成本"(美元线)。**只有两条线同时向下,才叫健康复利**;人力线降而美元线升,是"用钱买自主率"的伪复利。

## 3. ModelBinding auto-by-complexity 的成本治理:谁拦"什么都标高复杂度→全上 Opus"?

**场景**
绑定规则"复杂→Opus"。判"复杂"的主体——无论是 agent 自评、owner 配置、还是 §4.2 第 5 类"该类升 Opus"的复利规则——**都面临一个单向激励**:标成"复杂"上 Opus,质量护栏更好过、自主率更高、被复审打回的概率更低,而**成本代价不落在任何人的考核里**(§6 看板无成本)。于是:用户上报的普通单标"复杂",竞品雷达的自规划单标"复杂",连 file 类的简单分类都"为了稳妥"上 Opus。Opus 占比从 30% 漂到 70%,触发 §1 的 $11M/月情景。

**严重度:P0**
这是 §1(成本量级)的**根因机制**,也是 §2(复利反向)的放大器。DESIGN 全文对质量是"成对护栏"(§5.1 吞吐↔护栏强绑,护栏破阈一票否决),**对成本却完全没有对偶物**——这是设计哲学上的不对称漏洞。

**第一性原因——护栏的不对称。**
DESIGN §5 的核心洞察是"你度量什么,agent 就把什么推到极致(含退化方式),Goodhart 是工程事实"。设计者**为质量/吞吐造了完整的成对护栏**(§5.1 整张矩阵),却**对自己引入的成本维度只字未提**。结果:
- "升 Opus" 是一个**质量↑、自主率↑、成本↑** 的动作;
- 系统**奖励前两者、不惩罚第三者**;
- 按 §5 自己的 Goodhart 定律,**agent 必然把"升 Opus"推到极致**。
"复杂"没有操作化定义(§9 是自然语言),于是"标复杂"成了零成本的提分后门——**这正是 §5.4 "刃B 绝不能变成绕过护栏的刷分后门"的成本版本,但没人为成本设这道后门防线。**

**ModelBinding 的双向治理缺口(谁能升、谁能降、按什么真值)**
- **升的一侧无配额**:没有"Opus 预算池"或"高复杂度配额",任何主体可无限标"复杂"。
- **降的一侧无机制**:没有任何机制做反向 promote——"这类 Opus 产出其实 Medium 也能过"应当**降级**省钱,但 §4.2 第 5 类只有"medium 反复被改写→升 Opus"的**升级**路径,**没有对偶的"Opus 从未被改写/质量富余→降级"路径**。绑定是单调棘轮,只升不降。
- **真值缺位**:判"复杂"靠谁?没有"用便宜模型试跑、质量达标就不升级"的 challenger/影子降级评估。

**必须解决项 + 建议成本护栏指标**
1. **`opus_binding_rate` + 强制成对护栏**:仿 §5.1 把它和质量绑成一对——**升 Opus 必须附带"该绑定带来的质量增量证据"**(shadow 对比:Opus vs Medium 在该类上的护栏通过率差)。**无质量增量证据的 Opus 绑定不予通过**(成本版的"吞吐不得单独变绿")。
2. **Opus 配额/预算池 + 按 VaR 分配**:高 VaR/受保护边界(auth/payment/public_release)优先用 Opus,长尾低危**默认 Medium**,升级需理由(对偶 §6 的 anti-rubber-stamp)。
3. **`model_downgrade_candidates`(降级候选)看板 + 反向 promote 通道**:周期性影子降级评估——把 active 的 Opus 绑定用 Medium 影子跑,质量护栏持平则**自动建议降级**。让 §4.2 的绑定调整**双向**,棘轮变弹簧。
4. **绑定 provenance 带成本影响面**:§9 已要求配置改动"影响面预估",**补一列"成本影响面"**(改这条绑定,fleet 里多少在跑实例的单位成本变化)。owner 改绑定时看见美元后果。

## 4. 人力 $ vs 自治 $:审批人/owner 人力成本 vs 自治省下成本,盈亏平衡点

**场景**
系统稳态需要:工程师审 escalation、owner 评审 promotion/宪法级改动(§4.6① 四眼)、PM 盯护栏+做复审抽样人审(§5.2)。这些人力是**真金白银的固定/半固定成本**。问题:**在什么自主率/复利成熟度下,"自治省下的人力成本"才真正大于"运行自治本身所需的人力+算力成本"?DESIGN 从未计算这个盈亏平衡点。**

**费米估算(双侧损益表)**

*人力侧(用量级,单价应由运营填真实薪资)*
- 资深工程师/owner 全成本 ≈ **$60–$80/工时**量级(含 overhead);PM/QA 同量级。
- 假设稳态需 N 人运维 10k agent。**§6 截断 + §4 复利的全部意义,就是把 N 压到个位数。**
- 若 N=5(乐观)、月 ≈ 168 工时/人:**人力成本 ≈ $50k–$67k/月**。
- 若复利不收敛(§13 R-6,distinct 决策类发散),N 随规模重新爬升,**人力线性崩盘**——这是设计反复警告的失败模式。

*自治侧(§1/§2 已估)*
- 推理算力:**$1.8M/月(中档)**。

> **这一笔对比本身就是结论:中档下,算力成本(~$1.8M/月)比人力成本(~$50k–$67k/月)高出 ~30×。**

**这是整份审查最颠覆设计假设的一点。**
DESIGN 全文的可行性叙事是"**人力是会随规模崩盘的稀缺成本,自治是解药**"(§0 刃B、§4.1)。但费米一算:

> **在 10k 规模下,主导成本根本不是人力,是算力。** 设计把全部火力对准"省人力"(复利机、截断、批量),却对真正的大头——**算力**——没有任何治理。这是**优化错了成本项**:在拼命省一个 $50k 的项,同时放任一个 $1.8M(可失控到 $11M)的项裸奔。

**盈亏平衡点的真实形态(被设计忽略的那个)**
设计默认的盈亏平衡是"自治省的人力 > 自治需的人力"(一个**人力 vs 人力**的比较,复利曲线走平=退化为线性人肉运维)。但真实的单位经济学盈亏平衡是:

```
单位闭环总成本(自治) = 人力_自治 + 算力_自治 + 复利培育摊销
                  必须 <
单位闭环总成本(基线/纯人) = 人力_纯人
```

**当算力_自治 ≈ 30× 人力时,这个不等式可能根本不成立**——哪怕自主率做到 99%,只要每个闭环烧的 Opus 足够多,自治版**单位成本可以高于纯人版**。设计从没验证过这个不等式。

**严重度:P1**(机制级 P0、但绝对量级上算力远超人力,故"人力盈亏点"本身降为 P1;真正的 P0 是它**揭示了优化目标错位**——这点喂给 §6)

**第一性原因**
DESIGN 把"成本"**隐式等同于"人力成本"**(§0「随规模线性增加人力→商业崩盘」)。这个等式在"人贵、算力近免费"的旧世界成立,在"10k agent 烧 Opus"的新世界**反转**了。整套盈亏平衡叙事建立在一个未经检验、且很可能已经过时的成本结构假设上。

**必须解决项 + 建议成本护栏指标**
1. **`fully_loaded_cost_per_closed_loop`(满载单位闭环成本)= (人力 + 算力 + 复利培育摊销) / 闭环数**,**人力与算力分项可见**。这是判断盈亏平衡的唯一正确分母(详见 §6)。
2. **`autonomy_economic_breakeven`(自治经济盈亏点)**:画出"自治版单位闭环成本 vs 纯人基线单位闭环成本"的对照线。**自治版没有压到基线之下,就不能宣称"自治在省钱"**——只能宣称"自治在省人/扛规模",这是不同的主张,不能混淆。
3. **人力成本也要进账**:把审批/评审/抽审的**人时**计入单位闭环成本(不只是算力),否则盈亏平衡是假的。
4. **冷启动容量预算用美元、不用人数**(补 R-5/R-7):过渡期的人力缺口要标成"$/月额外人力 + $/月额外 shadow 算力",一起进预算。

## 5. 失败 / 重试 / 回滚成本:跑飞、loop 重试、坏规则回滚的算力浪费

**场景**
agent 跑飞(死循环/反复调工具/上下文爆炸)、loop 因 error 进 §3 状态机的 `error` 态后重试、坏规则被 §4.3 auto-demote / 一键回滚、§5.1 "合入后 14 天回滚"触发的返工。**每一次失败/重试/回滚都是已花算力的沉没 + 重做算力的追加,且若该 loop 绑了 Opus,沉没的是最贵的算力。** DESIGN 把回滚/drop **当成质量护栏的"信号源"**(§3「`closed_dropped/rolled_back` 反哺护栏指标」),**却从没把它当成"成本黑洞"来度量。**

**费米估算(浪费占比)**
- 设 agentic 任务返工率/重试率 ≈ **10–25%** 量级(agent 跑飞、gate 打回、reopen 都算)。
- 则**纯浪费算力 ≈ 总算力的 10–25%**:中档下 = **$180k–$450k/月**纯浪费,只为"做了又推翻"。
- **跑飞的放大效应**:一个没有 token 上限的 agent 陷入工具调用死循环,可在数小时内单独烧掉远超正常 loop 数量级的 token——R-2 已点出"agent 跑飞如何强制夺权"的 SLA 未定,**但只从控制权角度看,没从"夺权前它烧了多少钱"角度看。** 夺权 SLA 同时是个**成本止损 SLA**。

**坏规则回滚的复利级浪费**
§4.3 坏规则 auto-demote 后,它在 active 期间**自动解决**的那批案例,可能需要 reopen/返工——这是**复利机自己制造的回滚成本**,且因为是"自动解决",量可能很大、很隐蔽。R-3 盯的是"规则把错的自动放行"的**质量**尾部风险,**没盯它的成本尾部**:一条坏规则在被 demote 前自动放行 N 个错误闭环,返工成本 = N × 单位返工算力。

**严重度:P1**
绝对量级($180k–$450k/月浪费)显著但非生死级;真正的隐患是**它完全不可见**——回滚被当信号、不被当成本,跑飞没有 token 止损,坏规则的返工算力无人追溯。是个持续失血点。

**第一性原因**
DESIGN 的失败处理全部从**质量/控制**视角设计(护栏信号、夺权、auto-demote),**成本视角整体缺席**。"一次失败花了多少算力"从来不是一个被记录的量。沉没成本不可见 → 无人有动力降低失败率来省钱。

**必须解决项 + 建议成本护栏指标**
1. **`wasted_compute_ratio`(浪费算力占比)= (失败/重试/回滚/被 demote 规则返工)消耗的 token / 总 token**,看板常驻 + 红阈。
2. **单 loop / 单 agent token 硬上限 + 熔断**:超限自动暂停并升级人工(对偶 R-2 的夺权,但触发器是**成本**不只是时间)。**这是防"单个 agent 烧穿账单"的最基本止损,设计完全没有。**
3. **`cost_to_demote`(坏规则止损成本)**:一条规则从"精度掉阈"到"完成 demote+返工"之间自动放行错误闭环的累计返工成本。并入 §2 的 `net_dollar_savings_per_rule`(坏规则的负节省要被记账)。
4. **夺权 SLA(R-2)补成本维度**:定"最大让权时延"时,同时定"跑飞 agent 的最大可容忍烧钱额",取先到者触发强制 kill。

## 6. 度量看板缺成本维度:会不会优化出"又自主、又高质、但巨贵"的系统?

**场景**
北极星 = **受质量约束的自主率 ⊗ 决策杠杆**(§0/§5)。§5 整章把吞吐和质量护栏成对绑死,§5.2 复合健康灯三色判定**全部由"主指标方向 + 质量护栏阈"决定**。**成本不在北极星里,不在健康灯里,不在 §5.1 的吞吐↔护栏矩阵里,不在任何一盏灯里。** 运营一年:自主率 95%、护栏全绿、健康灯长亮绿,**月账单 $11M**,财务在另一个房间尖叫,而工作台显示"系统非常健康"。

**这是本次审查的"元发现",其他五条都收敛到这里。**

**严重度:P0(本审查的头号 P0)**
因为它是个**结构性失明**,不是某个指标缺失。按 DESIGN §5 自己立的法——"**你度量什么,agent 就把什么推到极致**"——**不度量成本,就等于命令整个系统把成本推到极致**。系统会忠实地优化出一个"自主率×质量"的局部最优解,而这个解在成本维度上**可以任意贵**,因为成本是它的自由维度。§5.4 警告刃B 不能变成"绕过质量护栏的刷分后门";**而整个系统,本身就是一个绕过成本护栏的刷分后门——因为成本护栏不存在。**

**第一性原因**
北极星的两个因子(自主率、决策杠杆)**都是"分子"性质的量**(做了多少、省了多少人次),**没有一个"分母/代价"性质的成本量**。一个只奖励产出、不计入代价的目标函数,数学上必然把代价推向无穷——这不是 bug,是目标函数定义不完整的**必然解**。DESIGN 极其严谨地为"质量代价"建了护栏(怕刷分),却对"经济代价"完全免疫——因为设计者的心智模型里,"代价"=人力,而人力已被复利机"解决"了(见 §4 揭示的错位)。

**必须解决项 + 建议成本护栏指标(本条 = 全文收口)**
1. **北极星升级为三因子**:**受质量约束的自主率 ⊗ 决策杠杆 ⊘ 单位闭环成本**。第三个因子是**除数**(代价项),让目标函数"在成本约束下"最大化产出,而非无约束最大化。
2. **健康灯加第四条铁律(成本破阈一票否决)**:仿 §5.2「护栏破阈一票否决」——`cost_per_closed_loop` 或 `blended_cost_per_1M_tok` 破红阈 ⇒ 健康灯**不得为绿**,无论自主率多漂亮。**质量护栏和成本护栏必须对称,这是 §5 哲学的逻辑闭包。**
3. **看板加"经济墙"**(对偶 §5.1 的"吞吐↔护栏对照墙"):单位闭环成本趋势、blended 单价、Opus 占比、浪费算力占比、复利净美元节省、自治 vs 基线盈亏对照——**六条成本线常驻一屏**。
4. **§5.5 self-trade 加成本审计**:"自规划提单→自己解"的自我交易闭环,不只查质量永动机,**还要查"自己给自己派活烧 Opus"的成本永动机**——self-trade 闭环的单位成本要单独抽审。

## 结尾:Top 3 经济 P0 + 单位经济学裁决

### Top 3 经济 P0

**P0-经济-1 · 看板成本维度全缺失(§6,元问题)。**
北极星与健康灯只有"自主率×质量",没有任何成本项。按设计自己的 Goodhart 铁律,不度量成本 = 命令系统把成本推向无穷。**必须把"单位闭环成本"升为第三北极星因子 + 加"成本破阈一票否决"。** 其余所有经济问题都是这一条的下游。

**P0-经济-2 · ModelBinding 成本护栏不对称(§3,根因机制)。**
质量有完整成对护栏,成本一个都没有;"标复杂→上 Opus"是零成本提分后门,而 30% 的 Opus 调用吃掉 ~85% 账单(§1)。绑定是只升不降的棘轮。**必须给 Opus 绑定配预算池 + 强制"质量增量证据"成对护栏 + 开反向降级通道。**

**P0-经济-3 · 复利机净账可能为负(§2,地基松动)。**
复利机只优化"省人次",从不把算力放进自己的损益表;shadow 双倍/K 倍算力 + 永久残留抽审人审,使它在长尾低危类别上很可能"省人力的同时更烧钱"。这动摇了"复利=唯一可行性通道"的全文地基。**必须用 `net_dollar_savings_per_rule`(带置信下界)给每条规则算美元净账,负节省的规则不予 promote;复利曲线必须人力线、美元线双线同降才算健康。**

### 一句话裁决

> **存疑(conditional / not-yet-viable)。**
> 这个设计在**扛规模、省人力、防刷分**上是成熟的;但它建立在一个已经过时的成本假设上——"代价 = 人力"。费米一算,10k 规模下主导成本是**算力(~$1.8M/月,失控可达 ~$11M/月,约为人力的 30×)**,而设计对算力**零治理、零度量、零护栏**。在补齐"单位闭环成本"北极星因子 + ModelBinding 成本护栏 + 复利净美元损益表(上述三条 P0)之前,**这个系统单位经济学上既未被证明可行,也存在被证伪的真实路径**——它完全可能优化出一个"又自主、又高质、但单位成本高于纯人基线"的系统,那在商业上和不可行是一回事。**不判"不可行":因为三条 P0 都是可补的度量/治理缺口,不是机制死结——补上即可重判。**

### 我建议"单位闭环成本"该怎么定义

```
单位闭环成本 cost_per_closed_loop
  = ( 推理算力成本            // 该闭环全生命周期所有 agent 的 token×单价,Opus/Medium 分项
    + 失败/重试/回滚浪费算力   // 沉没 + 返工(§5),按闭环摊
    + 复利培育摊销            // shadow/canary 预测推理 + 精度抽审,按受益闭环摊(§2)
    + 人力成本               // 审批/评审/复审抽样的人时 × 满载薪资(§4),分项可见
    )
  /  成功闭环数(merged + closed_done,排除 dropped/rolled_back)
```

四条配套铁律(让这个定义"咬得住"):
1. **分母只数真成功**:drop/rollback 不进分母(它们进 §5 的浪费分子)——否则"刷闭环数"能稀释成本,又一个 Goodhart 后门。
2. **人力与算力必须分项**:因为 §4 揭示二者量级差 ~30×,合并会掩盖"在省小头、放任大头"。
3. **全部带样本量 + 置信下界**(沿用 §4.6③):摊销与反事实项天然有不确定性,禁止用点估值充当北极星。
4. **入北极星做除数、入健康灯做一票否决阈**:让它和质量护栏完全对称——这是 §5 设计哲学的逻辑闭包,也是把"存疑"翻成"可行"的开关。
