## 0. 一句话结论(先给裁定,后给论据)

**这份设计在「刃A·路由」上确实活了;但 §4「刃B·复利机」目前是一张设计精美的电路图,核心的几根线没接上 —— 它把 C1 的人工瓶颈从「批 escalation」搬到了「批规则 + 喂 shadow 标注」,并没有从数学上消灭它。在「谁评审规则、规则冲突听谁的、active 规则的精度怎么算真值」这三件事定下来之前,刃B 不能宣称成立。** 详见 §1。

R1 把「刃B 缺席」判为最大盲区,R2 的回应方向对、骨架对,但**收敛动作本身引入了三个新的二阶失效模式**(复利机评审瓶颈、规则冲突治理真空、shadow 期负载不降反升),其中前两个够得上 P0,且 §13 未决风险表**漏了**。

## 第一性复盘:刃B 要成立,数学上必须同时满足的三个条件

在开火前先把「复利机数学成立」拆成可证伪的命题。DESIGN.md §4.1 宣称把人工负载从 O(n) 弯折为 O(distinct 未解决决策类),并称这是「几个人管几千 agent 在数学上成立的唯一通道」。要让这句话为真,**必须同时**满足:

- **(C-a) 规则的「生产成本」必须次线性。** 即:把一次决策固化成一条可信规则所需的**人工**,必须远小于它未来消除的人工总量,且这个生产动作本身不能是 O(规则数) 的人工。
- **(C-b) 规则的「维护成本」必须有界。** 即:N 条规则共存时,治理它们(冲突、过期、漂移、抽审)的人工不能随 N 线性增长,否则只是把运维对象从 loop 换成 rule,瓶颈原地不动。
- **(C-c) 规则的「真值」必须可廉价获得。** 即:你必须能持续知道一条 active 规则「自动放行的那些案例,人到底会不会同意」——否则 precision 是个无法计算的数字,刃B 就是开环刷分。

**DESIGN.md §4 把 (C-a)(C-b)(C-c) 全部默认为已解决,但文本里没有任何一个机制真正闭合它们。** 下面逐条压。

## 攻击矩阵

### 1. §4 复利机:解了刃B,还是把瓶颈搬了个家?

#### R2-A · 「谁评审 Promotion」是个黑洞 —— 复利机大概率把 C1 的 O(n) 瓶颈从「批 escalation」原地搬到「批规则」 — **P0**

**第一性原因.** DESIGN.md 通篇定义了 Promotion 的**生命周期**(shadow→canary→active)、**度量**(precision/coverage)、**载体**(`<PromoteToRuleAction>` 挂在每张决策卡上),却**从头到尾没有定义一个角色去评审「这条规则该不该被提升」**。把 §1 角色表摊开看,这个洞更刺眼:

- PM「**不批合入、不改资产**」(裁判≠运动员)—— 那 PM 显然不批规则。
- 工程师在单个 escalation 上 approve/reject —— 但一条 auto-approve 规则的影响面是**「这一整类未来所有同款 escalation」**,等于一条小型基线。让在控工程师顺手 promote,等于让运动员**给自己发「以后这类我都不用看了」的免审金牌**。
- §9 说「改基线=改宪法,走轻量评审」。Rule(尤其 `blueprint_patch`/`gate`/`model_binding` 三类)实质就是改宪法。**那 Promotion 到底走不走那个评审?DESIGN.md 没说。**

于是只有两种可能,且**都打脸 §4.1 的「数学上成立」**:

1. **每条 Promotion 都要人评审** → 你刚把「人工逐条批 escalation」(O(escalations))换成「人工逐条批规则」(O(distinct 决策类))。§4.1 自己承认未解决决策类的数量级是 O(distinct 决策类) —— 在一个有成千上万蓝图×gate×改动模式组合的系统里,**distinct 决策类本身就可能是几千几万**。这不是 O(1),这是「换了个名字的 O(n)」。复利机非但没减负,反而在每条 escalation 的处理流程上**新增了一个「要不要固化」的决策**(R2-C 会展开:shadow 期人还得继续判),**净增人工**。
2. **Promotion 不要人评审、由系统/在控人自助** → 那 §4.3 的「shadow-first + 抽审」是唯一闸门,而 R2-B/R2-D 会证明这两个闸门**关不严**。等于把「改宪法」的权力下放给了刷分压力最大的那个人,在最该有四眼的地方(§7 明明对 `asset:publish` 上了四眼原则)开了个后门。

**这是收敛动作亲手造的新瓶颈,R1 没碰过(R1 时复利机还不存在),§13 也没列。**

**必须解决项(R2-A).**
1. **明确定义 Promotion 的评审主体与触发条件**,且必须与 §7 的 RBAC/四眼对齐:auto_approve 低 VaR 类可走「shadow 自动达标即升」的轻通道;但 `gate`/`blueprint_patch`/`model_binding` 以及任何 scopeKey 覆盖高 VaR 档的规则,**必须走 §9 的宪法级评审 + 四眼**。把这条画进 §1 角色表和 §7,否则 RBAC 形同虚设。
2. **量化「规则评审」这条新流水线的负载**,把它和「escalation 评审」一起放进 §5.4 的 `<QueueFlowHeader>` 排队论模型 —— 否则你只是把一个隐性瓶颈从一个屏搬到另一个屏。**必须证明 `distinct 决策类的净增速率` × `每条规则评审成本` < 复利省下的人工**,这个不等式没写出来,§4.1 的「数学上成立」就是断言而非证明(见 R2-E)。

---

#### R2-B · 规则冲突 / 规则爆炸治理 = 真空。「规则的规则」在哪? — **P0**

**第一性原因.** §4.5 兴高采烈地说规则账本会「列出成千上万由决策固化的规则」。好 —— 那么第一性地问:**成千上万条 auto-approve 规则,scopeKey 必然互相重叠、嵌套、甚至矛盾时,系统听谁的?**

DESIGN.md 对这个问题**零字处理**。`Rule` interface 里只有 `scopeKey: string`,**没有 priority、没有 specificity、没有 precedence、没有 effective-dating 的冲突裁决**。具体崩法:

- **重叠**:规则甲「蓝图X+gateA+VaR≤t → 放行」与规则乙「蓝图X+任意gate+改动模式M → 升级人工」对同一个 escalation 同时命中,一个说放行一个说叫人。**听谁的?** 没有「more-specific-wins」或显式优先级,行为**不确定** —— 在一个自动放行真实合入的系统里,不确定 = 随机决定要不要叫人 = 直接击穿 §5.3 的 false-autonomy 防线。
- **矛盾固化**:工程师 A 在周一把「这类 auth 改动放行」固化成规则;工程师 B 在周三对一个长得几乎一样的 case 判了「这类必须升级」并也固化。两条规则**语义对立**,系统里同时 active。没有冲突检测,两人都不知道对方存在 —— 这正是 R1 的 D1「无共享学习、fork→drift」**在规则层的复活**:R2 用规则消灭了 Agent.md 的 fork drift,却**在规则集合里重新制造了一模一样的 drift**,只是换了个载体。
- **爆炸后的可治理性**:§4.4 的「规则精度」是逐条算的;但**系统级**的「这 8000 条规则合起来覆盖了哪些区域、留了哪些洞、有没有环、有没有死规则」——**没有任何聚合视图或一致性检查**。§4.5 的账本是个**列表**(又是 R1 P0-2 痛批的 O(n) 列表范式!),不是一个可治理的规则系统。规则账本本身在 1k 条规则时就重蹈 fleet 列表的覆辙。

**「规则的规则」(meta-governance:优先级体系、冲突检测、覆盖/空洞分析、死规则回收)在 DESIGN.md 里完全不存在。** 这违反了 §4 自己第一性定位里「让刃B 可见、可量化、**可治理**」的承诺 —— 可治理是吹的,治理机制没写。

**必须解决项(R2-B).**
1. **定义规则的冲突裁决语义**,至少包含:(a) specificity 排序(更具体的 scopeKey 优先)或显式 priority 字段;(b) **「升级人工」类规则永远压倒「自动放行」类规则**(fail-safe 默认:冲突时偏向叫人,不偏向放行);(c) 提升新规则时**强制冲突检测** —— 与现有 active 规则集做交集分析,命中矛盾则阻断提升并要求人裁决「以哪条为准」。这本身又是一个评审动作,必须并回 R2-A 的负载账。
2. **规则账本必须是可聚合的治理面,不是列表**(直接套用 §2 对 fleet 的 rollup 范式):按 scopeKey 维度 rollup、覆盖热力图、空洞/重叠/死规则(长期 0 命中或长期 0 抽审)告警、规则总数随时间曲线(规则数无界增长本身=退化信号)。
3. **给「规则集」一个版本与回滚边界**:不只单条 Rule 可回滚(§4.5 已有),整个规则集要能快照,以便「上周引入的某条规则群体性放行了坏东西」时一键回退到一致的旧状态。

#### R2-C · Shadow 期负载不降反升,且长尾规则永远毕不了业 —— 复利曲线可能「假性走平」 — **P1**

**第一性原因.** §4.3 的 shadow-first 机制本身藏着一个被乐观叙述盖住的代价:**shadow 模式要度量「规则会怎么判 vs 人怎么判」的一致率,这要求人在 shadow 期对这些 case 全部继续亲自判。** 也就是说:

- 一条规则从草案到毕业(active)的整个 shadow + canary 窗口里,**它消除的人工 = 0**(人还在判),而它**新增的人工 = 「决定要不要 promote」+「盯 shadow 一致率」**。复利的收益在毕业那一刻才开始,**转换期是纯负投资**。DESIGN.md 把这段成本完全隐去了 —— §4.4 的「复利曲线 = 每千 loop 所需人工决策数」在 shadow 期只会**上升或持平**,不会下降。
- 更狠的结构性问题:**一条规则要积累足够 shadow 样本才能毕业。高频决策类样本来得快、很快毕业;低频/长尾决策类样本来得慢,可能永远凑不够样本 → 永远卡在 shadow → 永远要人判。** 而恰恰 §6 已经说了,高频低危类**本来就能用 approve-all-similar 廉价批量**。于是复利机真正吃掉的,大多是「本来就便宜」的那部分;真正昂贵的长尾,它**够不着**。这会让复利曲线下降到某个位置后**假性走平**,而 §4.4 把「走平」直接定义为「退化成线性人肉运维」的告警信号 —— 设计可能会对着自己机制的固有上限狂报警。

**必须解决项(R2-C).**
1. **复利曲线必须把 shadow/canary 的「在途投资」单独画出来**(净人工 = 已毕业规则省下的 − shadow 期新增的),否则曲线会在冷启动期(§13 R-5)误导决策者以为复利无效而砍掉它。
2. **为长尾决策类定一条不同的路径**:低样本类不该靠「凑够 shadow 样本」毕业,而应允许「人显式声明置信、小样本直升 canary + 更高抽审比例」——把举证责任从「样本量」换成「人的显式背书 + 加重事后抽审」。否则刃B 对长尾无效,而长尾恰是人工成本的大头。
3. **区分「健康走平」与「触顶走平」**:复利曲线走平时,系统要能回答「是没有新的可固化类了(健康),还是剩下的都是固化不了的长尾(触顶)」——否则 §4.4 的告警语义是错的。

#### R2-D · 复利放大坏决策:§4.3 的缓解对「人系统性判错」无效 — **P1**(逼近 P0)

**第一性原因.** §4.3 防的是「**从单次决策学出的规则可能是错的**」(过拟合,§13 R-3 也是这么写的)。但复利机最危险的失效不是「单次偶然判错」,而是「**人系统性地、一致地判错**」—— 而 shadow-first 对后者**结构性失明**:

- shadow 的毕业标准是「规则判的 vs **人**判的,一致率 ≥ 阈值」。**如果人本身在这一类上系统性判错(比如对某类隐蔽的鉴权绕过,人 reviewer 一直误以为安全),那规则会高度一致地复制这个错误判断,一致率 100%,完美毕业,然后把这个错误以机器速度复利放行到全 fleet。** shadow 用「人」当 ground truth,而当人就是错误源时,这个闸门**自动失效**。这正是 R1 B2 警告的「自动驾驶为了脱离率好看而该接管不接管」的复利版 —— 现在是「人和规则一起、自信地、大规模地犯同一个错」。
- §4.3 的第二道闸「精度持续抽审」也是用**人**复审(§5.2 复审抽样池)。同一批可能系统性判错的人去抽审,**抽审命中率对系统性盲区同样为 0**。两道闸用的是同一个可能有偏的真值源,所以**不是两道独立闸,是一道闸的两个副本**。
- §13 R-3 把这个风险标为「shadow-first + 精度抽审 + auto-demote **缓解**」—— 这是**自评乐观**(见 R2-G):它缓解的是「单条规则的方差(偶然错)」,完全不缓解「真值源的偏差(系统错)」。把「降方差」当成「降偏差」来交账,是统计意义上的偷换。

**必须解决项(R2-D).**
1. **承认 shadow 的 ground truth 有偏风险,引入独立真值源**:对高 VaR 类规则,抽审不能只靠「原班人马事后看」,要接**下游、滞后、人之外**的信号当真值 —— 即 §5.1 已经定义的「合入后 14 天回滚/缺陷率」「reopen 率」。**规则精度必须与它放行案例的下游回滚率挂钩**,而不只是「人会不会当场点头」。这把「人当场同意」(易被系统性偏差污染)替换/补强为「事实上有没有出事」(难造假)。
2. **坏决策的复利「半径」必须有熔断**:一条 active 规则的下游回滚率一旦破阈,不仅 auto-demote 该规则(§4.3 已有),还要**追溯并复核它已经自动放行的历史案例**(blast-radius 回溯)——否则「demote」只停了出血,没处理已经放出去的坏合入。§13 R-3 的待办「盯尾部风险」太软,要落成「下游指标触发的强制回溯」。

### 2. 「自己给自己打分」:DESIGN.md 里的乐观 / wishful thinking 盘点

> R2 由编排者(持全量上下文)代行 worker agent 写成,§13 已诚实声明。但「诚实声明流程」≠「内容里没有自评夸大」。下面专挑「**说了等于做了 / 定义了等于实现了 / 缓解了等于解决了**」的地方。

#### R2-E · 「数学上成立」被反复断言,但等式从未写出 — **P1**

**第一性原因.** §4.1 原文:「这是『几个人管几千 agent』在**数学上成立**的唯一通道」。§0.1 北极星、§6 也都把复利当成让算术闭合的钥匙。但 **DESIGN.md 全文没有一个数字、一个不等式去支撑「数学上成立」**。R1 的 C1 是用**算术**(200–500 稳态积压 × 3–5 分钟 = 数十全职审批)把「不成立」**算出来**的;R2 要反驳它,**必须用同等的算术把「成立」算出来**,而不是用一个机制的存在去断言结论。

「O(n) 弯折为 O(distinct 决策类)」这句话偷偷假设了 distinct 决策类**远小于** n 且**增长收敛**。但这恰恰是**待证命题**,不是已知条件 —— 一个开放产品域里,新蓝图、新 gate、新改动模式持续涌入,distinct 决策类**可能本身就是发散的**。把待证的当已知,是典型的自评 wishful thinking。

**必须解决项(R2-E).** §4 必须补一个**量化论证**(哪怕是带假设的费米估算):给定 N loop、escalation 率 e、distinct 决策类增速 g、每类规则的生产+评审+维护成本 c_total、单条规则覆盖率 r,**写出「稳态人工 = f(以上参数)」并论证它在目标人力内有界**。否则把「数学上成立」**降级为「机制上可能收敛,参数与发散性待运营验证」** —— 这才是诚实的措辞(对标 §12 对护栏阈值用的「机制就位、参数待校准」)。

#### R2-F · precision / coverage / savedHumanDecisions 是**定义了字段**,不是**算得出真值** — **P1**

**第一性原因.** §10 的 `Rule` interface 把 `precision`、`coverage`、`savedHumanDecisions` 列为字段,§4.4 给了定义。但**字段存在 ≠ 真值可算**:

- `precision`(规则精度 = 规则自动解决时人会同意的比例):对 active 规则,人**已经不看了**(这就是 active 的意义)。你只能从**抽审样本**估计 precision,而样本估计有置信区间。DESIGN.md 把 precision 当成一个**确定的标量**展示在 `<RuleCard>` 上,**没有置信区间、没有样本量**。一条只抽审过 5 次的规则显示「precision 100%」,会给人灾难性的虚假信心。这是把「采样估计」冒充成「总体真值」。
- `savedHumanDecisions`(决策杠杆累计):这是个**反事实**(counterfactual)—— 「如果没有这条规则,人本会介入多少次」。反事实不可直接观测,只能在强假设下估计。DESIGN.md 把它当成可累加的硬计数,**等于把一个估计量供成了 KPI** —— 而 §5 自己刚痛斥过「把可刷的数字供成北极星」(B1)。**决策杠杆这个北极星因子,本身就是个可被乐观估计刷高的数字。**

**必须解决项(R2-F).** 所有复利度量必须带**采样不确定性**:`<RuleCard>` 显示 precision 时强制并列「样本量 + 置信下界」,**用置信下界而非点估计驱动 auto-demote/promote 决策**(小样本高估时偏保守)。`savedHumanDecisions` 必须标注为「估计值」并写明反事实假设,**禁止它单独作为北极星展示**(必须与 false-autonomy、下游回滚率成对 —— 套用 §5 对一切吞吐指标的同款铁律)。

#### R2-G · §13 把「缓解」记成「现状」,弱化了残留严重度 — **P2**

**第一性原因.** §13 表格的「现状」列普遍用「**机制就位 / 已缓解 / 语义已定**」收尾(R-2「语义已定 SLA 未定」、R-3「shadow + 抽审 + auto-demote 缓解」、R-4「保守约束缓解」)。这种措辞把**做了一半**说成**做了**。R2-D 已证 R-3 的「缓解」对系统性偏差无效;R-4 的「四元保守约束」R1 自己就标了「残留风险见 §13」,是个自指循环(§4.3 指向 §13,§13 R-4 又指回 §4.3),**两边都没真正定 scopeKey 划错的可审计/可追溯回滚的具体机制**。「自己引用自己当解决方案」是收敛文档里最隐蔽的一种 hand-waving。

**必须解决项(R2-G).** §13 每条的「现状」改为**冷酷的二元判定**:`机制设计完成 / 机制未设计`,并把「参数待定」「真值源待定」「评审主体待定」从「现状」挪到「待办」并标 P 级。**禁止用「缓解」「就位」「已定」模糊掉「关键子机制尚不存在」的事实。**

---

### 3. §13 五个未决风险:漏了真正的 P0

**判定:§13 列的 5 条(R-1 护栏阈值 / R-2 接管 SLA / R-3 复利过拟合 / R-4 聚类边界 / R-5 冷启动人力)质量不错,但全部是「参数/标定/过渡」类(知道要做什么,只差校准)。它系统性地漏掉了收敛过程引入的「机制本身不存在」类 P0。** 缺口如下:

#### R2-H · 漏 P0:「谁评审规则 + 规则冲突治理」未列入未决风险 — **P0**

即 R2-A + R2-B。这是 R2 新增机制(复利机)的**一阶设计缺失**,不是二阶参数问题,严重度高于表内任何一条,却**完全不在 §13**。**这是「自己写、自己审」最典型的盲区:你最不会去质疑自己刚发明的、最得意的那个机制。** 复利机是 R2 的明星,于是它的地基洞反而没人记进风险表。

#### R2-I · 漏 P0/P1:收敛引入的三个新机制各自的二阶问题,§13 一个都没收录

R2 为了解 R1 的 P0,新引入了三套机制,每套都有自己的二阶失效,§13 **一条都没列**:

| 新机制(R2 为解 R1 而加) | 二阶失效模式(本终审新发现) | 严重度 |
|---|---|---|
| **复利机 §4** | 评审瓶颈(R2-A)、规则冲突真空(R2-B)、shadow 期负投资+长尾够不着(R2-C)、系统性偏差被复利放大(R2-D) | P0 / P1 |
| **RBAC + 四眼 §7** | 四眼原则**只挂在 `approve:protected`/`asset:publish`**,**没挂在 Promotion 上**(R2-A.1)——最该四眼的「批量免审金牌」反而没四眼。且四眼在「几个人管几千 agent」的稀缺人力下,**本身就是 C1 瓶颈的放大器**(高 VaR 决策现在要 2 个稀缺的人) | P1 |
| **队列即流量系统 §5.4** | `<QueueFlowHeader>` 只建模了**escalation 流**,**没建模规则评审流、shadow 标注流**(R2-A.2)。多个队列互相耦合(降派发速率会同时饿死复利的样本供给 → 越想止血越固化不了规则),**没有跨队列的协调策略** | P1 |

**§5.4 那条「加速复利以从源头降流入」的建议在这里露出一个致命耦合**:复利要毕业需要 shadow 样本,样本来自 escalation 流量;而队列告警的另一个建议是「降派发速率」(减流量)。**两个建议方向相反 —— 减流量会饿死复利的样本供给,让规则更难毕业,从而更依赖人,从而更需要减流量。** 这是一个 R2 自己埋的正反馈死锁,§13 完全没意识到。

**必须解决项(R2-H/I).** §13 必须**新增**至少:R-6「Promotion 评审主体 + 四眼覆盖未定」(P0)、R-7「规则冲突裁决 + 规则集治理机制不存在」(P0)、R-8「多队列耦合 / 复利样本供给 vs 限流的正反馈死锁」(P1)。把它们从「我没想到」变成「我明牌承认还没做」。

## 终审判决 + Top 3 硬前置

### 终审判决(一句话)

> **这份设计的「刃A·路由」已经活了,可以据此做高保真;但「刃B·复利机(§4)」还没活 —— 它在纸面上闭环、在数学上未闭环,且收敛动作亲手埋了「无人评审规则、规则冲突无裁决、shadow 期负投资」三个新 P0/P1,其中两个 P0 连 §13 风险表都没收录。结论:可以进高保真 mockup 与 §A 路由侧的实现准备;但 §4 复利机在补齐下面 3 条硬前置之前,不得进入实现,也不得在任何对外材料里宣称「数学上成立」。**

一句话裁定:**「半活」—— 路由侧绿灯进实现,复利侧黄灯卡 P0,带 3 条硬前置返工。**

### Top 3 硬前置(进实现前必须先定,缺一不可)

1. **【定主体】Promotion 的评审主体 + 四眼覆盖(解 R2-A / R2-H)。**
   明确写进 §1 角色表与 §7:哪类 Rule 走 shadow 自动毕业的轻通道,哪类(`gate`/`blueprint_patch`/`model_binding`/任何覆盖高 VaR 档)**强制走 §9 宪法级评审 + 四眼**;并把「规则评审」这条新流水线的负载并入 §5.4 排队论模型。**不定这条,复利机就是个无人把守的「批量免审金牌发放机」,RBAC/四眼全部形同虚设。**

2. **【定仲裁】规则冲突裁决语义 + 规则集治理面(解 R2-B / R2-H)。**
   定义:specificity/priority 排序 + **「升级人工」永远压倒「自动放行」的 fail-safe 默认** + 提升时**强制冲突检测**;并把规则账本从 O(n) 列表升级为可 rollup 的治理面(覆盖/空洞/死规则/冲突告警 + 规则集快照回滚)。**不定这条,§4 就在规则层复活了 R1 D1 的 fork-drift,且自动放行行为不确定 = 直接击穿 false-autonomy 防线。**

3. **【定真值】active 规则的精度真值源 + 带不确定性的度量(解 R2-D / R2-E / R2-F)。**
   规则精度必须挂**下游、滞后、人之外**的信号(合入后回滚率/reopen 率)当独立真值,而非只靠「原班人马当场点头」;所有复利度量(precision/savedHumanDecisions)强制带样本量 + 置信下界,**用下界驱动 promote/demote**;`savedHumanDecisions` 标注为估计值且禁止单独当北极星;并把 §4.1 的「数学上成立」**降级为带费米估算的「机制上可能收敛、发散性待验证」**。**不定这条,刃B 就是用可能有系统性偏差的人当唯一裁判、把采样估计冒充总体真值的开环刷分器 —— 正是 §5 自己最痛恨的那种「可刷的北极星」。**

---

### 给编排者 / PM / 设计的一句话

R1 之后你们把刃A 打磨得很扎实,刃B 的方向也找对了 —— 但你们**爱上了自己刚造的复利机**,于是把它最薄弱的三块地基(谁评审、听谁的、真值哪来)当成已解决,还顺手让它躲过了 §13 的风险点名。**自己写、自己审的最大代价,就是最得意的那个新机制最不会被自己质疑。** 把上面 Top 3 硬前置当成 §4 进实现的闸门:在它们落成之前,复利机是一张漂亮的电路图,不是一台会发电的机器。
