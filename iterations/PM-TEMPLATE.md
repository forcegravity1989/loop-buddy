# 项目管理模板 · 五角色五阶段环(以 WorkflowHub 自身为样例)

> **怎么用这份模板**:把你的项目当作一个要"绕环五圈"的产品。每绕一圈,依次让五个角色各司其职一轮,环闭合后(Ops 回流 Prototype)开下一圈。每圈的证据是真实可跑的产物 + 真实度量,不是 PPT。
>
> **样例**:WorkflowHub 自己就是用这份模板建的——下表的"样例产物"列,是本仓库 25 轮迭代里该阶段真实落地的代码/文档/测试,可逐一核对。

---

## 环总览(线闭成环)

```
Prototype(原型·求真)→ Build(构建·求成)→ Optimize(优化·求简)
   ↑                                           ↓
 Ops(运维·求稳)  ←─── 回流 ───  Growth(运营推广·求增)
```

| 阶段 | 角色 | 方法论 | 心法 | 样例产物(WorkflowHub iter) |
|---|---|---|---|---|
| 原型 | 原型师 · Prototyper | 假设驱动探索 | 求真 | 每轮开头的"假设/DoD"(iters 1–25 的原型师段) |
| 构建 | 构建师 · Builder | 规格驱动交付 | 求成 | `workflow_run` 表 / analysis 模块 / 自驱循环(iters 1,7–18) |
| 优化 | 优化师 · Optimizer | 度量驱动打磨 | 求简 | clippy 修复 / 幂等 settle / median 抗离群(iters 1,2,10) |
| 运营推广 | 运营推广师 · Grower | 增长实验 | 求增 | 仿真器 / 场景聚类 / 推荐(iters 15,17,21,22) |
| 运维 | 运维师 · Maintainer | 可靠性工程 SRE | 求稳 | 迁移守卫 / 事件可订阅 / 持久化重开还原(iters 1,4,5) |

---

## 每个阶段做什么(模板 + 样例)

### 1. 原型师 · 假设驱动探索 · 求真
- **输入**:现状(真实可观测,不靠猜)+ 一个待验证的假设。
- **动作**:把现状和目标的差距,压缩成**一句话假设 + 一个可证伪的 DoD(完成定义)**。
- **出口**:DoD 写在迭代日志里,后面三个角色都对它负责。
- **反模式**:跳过假设直接写代码 = 没有靶子射箭。
- **样例**(iter 1):假设"没有成败颗粒度则优化是瞎猜"→ DoD"每次运行落一条 append-only 记录带 status/duration/trigger"。

### 2. 构建师 · 规格驱动交付 · 求成
- **输入**:DoD。
- **动作**:按既有架构模式落地(不发明新约定);类型先行(非法状态不可表达);写下来就是规格。
- **出口**:代码 + 迁移(若涉 schema)+ 能跑的最小验证。
- **反模式**:为单一需求发明第五种模式。
- **样例**(iter 1):`RunStatus` enum 封口 + `workflow_run` 表 + `NewWorkflowRun` 参数结构体(对齐 `New*` 模式)。

### 3. 优化师 · 度量驱动打磨 · 求简
- **输入**:构建师的产物。
- **动作**:跑门禁(fmt/clippy/test);用度量发现问题(中位抗离群、幂等、容错);删冗余。
- **出口**:门禁绿 + 一个"为什么这样更简/更稳"的度量理由。
- **反模式**:没有度量的"重构"= 凭感觉。
- **样例**(iter 2):成功率用中位耗时(非均值)——单个慢 run 不污染典型成本;无 settled 时 `None`≠0%。

### 4. 运营推广师 · 增长实验 · 求增
- **输入**:优化后的产物。
- **动作**:对着真实场景验证"它服务于谁、值多少";一句话能说清价值;给下一步种数据。
- **出口**:场景验证 + 价值一句话 + 回流数据已种。
- **反模式**:造了没人用的功能。
- **样例**(iter 22):演示一个失败工作流被诊断→修复→证明改善(29%→57%,A/B Improved)——价值的可运行证据。

### 5. 运维师 · 可靠性工程 SRE · 求稳
- **输入**:运营推广师验证过的产物。
- **动作**:持久化安全(迁移守卫、幂等、重开还原);可观测(事件可订阅);**回流**——把这一圈的产物/数据明确交接给下一圈的 Prototype。
- **出口**:重开还原通过 + 回流声明(下一圈 Prototype 要吃什么)。
- **反模式**:上线即失忆,没有可观测、没有接力。
- **样例**(iter 4):`cron_task_id` 用 `add_column_if_missing` 守卫(真实事故教训:`CREATE TABLE IF NOT EXISTS` 不加列);事件 `OptimizationCycleReported` 可订阅。

## 每圈(迭代)的最小清单

每轮迭代,五个角色各一行,门禁必跑:

```
[ ] 原型师:假设 = ___;DoD = ___
[ ] 构建师:落地的代码/表/类型(对齐既有模式)
[ ] 优化师:fmt ✓ / clippy -D warnings ✓ / test ✓ ;度量理由 = ___
[ ] 运营推广师:场景验证 = ___ ;价值一句话 = ___
[ ] 运维师:迁移守卫 ✓ / 幂等 ✓ / 重开还原 ✓ ;回流给下一圈 = ___
```

> **诚实约束**(贯穿五角色,不可妥协):
> 1. **绝不编造**:每个数字、每条证据都来自真实可跑的代码/测试/git log。Growth/Ops 不越过真实证据推进。
> 2. **无数据 ≠ 绿**:未知就是未知,不准用"绿色"粉饰"没测过"(镜像 `Signal::Unknown`)。
> 3. **破坏性永不自动**:自驱循环只自动应用正向/安全项;改内容/退役必须人工门控。

## 套用到你的项目

1. **定你的五阶段** —— 不一定是 Prototype/Build/...;但必须是"一个闭环",最后回流第一段。
2. **每圈五角色各一轮** —— 用上面的清单,日志记假设/DoD/度量/回流。
3. **门禁每圈跑** —— 哪怕门禁只是"能跑 + 一条断言",也要每圈绿。
4. **证据可核对** —— 每句"我做了 X"都能指向一个 commit / 一个测试 / 一个可跑的 example。

> 这份模板本身是 **WorkflowHub 用 25 轮迭代证明过的**:每轮的 `iterations/JOURNAL.md` 条目就是清单的实例。模板不是理论,是这套过程跑通的复盘。
