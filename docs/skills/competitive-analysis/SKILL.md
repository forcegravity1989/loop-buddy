# 竞品分析(competitive-analysis)

标配 Issue 三件套(**竞品分析** → 找指标 → 绑数据,plan/13 D8)里的第一
件,单次研究活(plan/13 D9),不是多轮对抗式评审——那种更重的形态留给
`plan/12` 的 Workflow 机制成熟后再接。本 Skill 的产出是「找指标」
(north-star-discovery)的输入:没有真实对标名单,「知道对标谁」这条产
品命题的第一个控制点就无从谈起。

## 何时用 / 前置条件

- 项目已建好,工作区是一个真实 git 仓(`workspace_path` 非空)。
- 读项目自身意图:项目的 `desc`(项目类型/一句话说明)、`benchmark`(创
  建流快答的对标对象)、`opportunity`(差异化机会)——这些字段是"用户已
  经知道的对标线索",本 Skill 的活是把它们坐实、查证、补全,不是无视它
  们另起炉灶。
- 若工作区根下已存在 `docs/competitive-analysis.md`(例如是修订而非首次
  起草),先读一遍,原地改写、不无理由推倒重来。

## 硬性约束(白纸黑字,plan/13 D9 · 绝不妥协)

> **绝不由幻觉填充对标事实。** 报告里的每一条对标信息——公司/产品是
> 谁、它的定位、它的用户规模、它的定价、它的北极星猜测——要么标注真实
> 检索来源(URL 或可查证的出处),要么明确标注 **「未核实/人工提供」**。
> 不允许存在第三种状态:一条看起来言之凿凿、却既没有来源标注也没有人工
> 提供标注的"事实"。宁可少写一条、宁可某个字段留白写"未核实",也不能
> 让训练记忆里的印象冒充成这次真的查过。
>
> **检索能力先探活,不通则如实降级**,不是"检索失败就摆烂不产出"。降
> 级路径是明确的操作步骤(见下一节),报告封面必须如实标注材料来源。

## 检索能力探活(先做这一步,不要跳过)

在写报告正文前,先确认这次执行器是否真的有联网检索能力——不要假设"应
该有"就直接编。判定依据(任选其一,越靠前越可信):

1. **本次会话里已经成功用检索工具查到过至少一条真实、可复核的信息**(有
   实际返回内容,不是工具调用报错或空结果)——通,继续走「检索可用路
   径」。
2. **本仓 `docs/skills/competitive-analysis/PROBE.md` 记录着一次近期
   (几天内)的真实探测结果为"通"**——可以参考它作为背景判断,但**不能
   替代本次会话自己的真实尝试**:检索能力受账号配额/网关抖动影响,今天
   不通不代表这次会跑通,反之亦然,一律以本次会话的真实尝试结果为准。
3. **本次尝试检索后拿到的是报错、拒绝、空工具列表**——不通,走「检索不
   可用路径」。

### 路径 A · 检索可用

正常检索、正常引用。每条来源写清 URL;summary/结论性判断(如"北极星猜
测")要说明是从哪条来源的哪部分推出的,不是复制粘贴,是有依据的推断。

### 路径 B · 检索不可用(明确操作步骤,不是含糊的"尽力而为")

1. 在活对应的 GitHub issue(竞品分析那张)下方留一条评论(或直接在
   `docs/competitive-analysis.md` 顶部写清楚),明确请求:**"本次执行器
   无联网检索能力,需要人工提供对标材料——请在此 issue 下贴对标产品的官
   网链接/定价页/公开数据(用户量、融资、评测文章等),或把资料文档直接
   放进工作区(如 `docs/inbox/` 下)。"**
2. 如果这一轮跑之前,issue 描述或工作区里**已经**有人工贴好的材料(用户
   评论里的链接、`docs/inbox/` 下的文件),直接使用——**只整理归纳**,
   不做未经检索的事实断言,不用自己的训练记忆去补人工材料里没写的空白。
3. 产出的报告封面(`docs/competitive-analysis.md` 顶部)**必须**如实标
   注:
   ```
   > 材料来源:人工提供,未经检索核实。本报告基于用户在 issue/工作区里
   > 提供的材料整理归纳,agent 未做独立联网核验。
   ```
4. 若这一轮既没有检索能力、也还没等到人工材料,**如实产出一份"占位 +
   请求"版本**:对标名单栏位写"待人工提供材料",不编造任何一家公司的名
   字或数据;PR 描述里说明"等待人工材料,后续可在同一分支追加提交"。空
   白诚实,好过用编造填满表格。

## 工作步骤

1. **探活**(见上节),确定走路径 A 还是路径 B。
2. **起草对标名单**:每家一行,写清楚"是谁"(公司/产品名 + 一句话定
   位)+ "为什么算对标"(和本项目在同一问题域/同一用户群/同一北极星候
   选上可比)。数量不强求,3-6 家是常见的够用规模;找不满就诚实写少,不
   凑数。
3. **各家北极星猜测**:基于路径 A/B 拿到的材料,推测每家对标产品自己在
   优化什么指标(留存?转化?活跃度?)。每条**必须**标来源(检索到的文
   章/官方数据页 URL,或"人工提供:<issue 评论链接/文件路径>",或诚实
   标"未核实——基于公开定位的合理猜测,无直接数据支撑")。
4. **差异定位**:我们 vs 每家对标,列出关键的取舍差——不是"我们更好"
   这种空话,是具体在哪个维度选择了不同的权衡(例如"对标 X 主打企业级合
   规,我们主打单人构建者的启动速度,因此不做团队协作面")。
5. **可借鉴打法**:从对标身上学到的、可执行的具体动作(不是"要重视用户
   体验"这类空谈)——例如"X 的 onboarding 用一次性清单代替长表单,值得
   在创建流里参考"。每条打法说清楚"抄哪一点""预期为什么对本项目有效"。
6. **落文件**:写进工作区 `<workspace>/docs/competitive-analysis.md`,
   四段顺序固定为①对标名单②各家北极星猜测③差异定位④可借鉴打法(见下
   方"输出契约"的完整样例)。路径 B 的封面标注(见上节步骤 3)放在文件
   最顶部,四段结构照写,只是内容里如实体现"未核实/人工提供"。
7. **交付**:改动落在活分支(`bw/issue-<n>`,由 `RunIssue` 在跑本活前
   已经切好),正常提交改动即可,提 PR 走执行器既有机制(跑完后
   `RunIssue` 自动尝试 `open_pr`);**执行器只会提 PR,合并永远是人手动
   作**。

## 输出契约

### `docs/competitive-analysis.md`

无固定 schema(人读文档,不被机器解析,和 `metrics-rationale.md` 同类
性质),但结构固定为四段、顺序不可打乱(后续「找指标」Skill 按固定顺序
读取)。路径 B 产出在文件最顶部另加一行来源标注(见上节)。

## 完整样例(可直接改写后使用,示范路径 A 的引用写法)

以下是一份结构合法的样例——虚构项目「个人知识库同步工具」,示范"每条
对标信息都标来源"的正确写法:

```markdown
# 竞品分析

## ① 对标名单

- **Obsidian Sync**(官方付费同步服务):本地优先的 Markdown 笔记 +
  官方端到端加密同步——和本项目同属"本地优先知识库同步"问题域,是最直
  接的对标。
- **Notion**(云端协作文档/知识库):虽然不是本地优先,但抢占同一批"个
  人知识管理"用户心智,是认知层面的对标,即使技术路线不同。
- **Logseq Sync**(开源大纲笔记 + 官方同步 beta):同属本地优先阵营,
  用户群高度重叠,直接功能对标。

## ② 各家北极星猜测

- Obsidian Sync:猜测其北极星是"付费订阅留存率"——来源:
  https://obsidian.md/sync 定价页把"跨设备无缝同步"作为付费转化的核心
  卖点,产品设计上高度聚焦这一体验的可靠性。
- Notion:猜测是"周活跃编辑用户数(WAU)"——来源:Notion 2025 年度博客
  文章 https://www.notion.so/blog/notion-2025-year-in-review 明确提到把
  "编辑频次"作为内部北极星披露。
- Logseq Sync:未核实——基于其开源社区的公开定位("数据主权优先"),
  猜测其北极星更可能是"数据不丢失率/同步成功率"而非增长类指标,但没有
  找到官方直接披露的数据支撑这条猜测。

## ③ 差异定位

- Obsidian/Logseq 都要求用户自行理解"仓库(vault)"概念才能用好同步,
  本项目选择"零配置默认同步",牺牲高级用户的目录自定义灵活性,换取新手
  的启动速度——这是"知道对标谁"之后主动选择的取舍,不是能力不足。
- Notion 靠云端协作取胜,本项目明确不做多人协作面(单人构建者场景),
  这条差异是范围收敛,不是尚未做到。

## ④ 可借鉴打法

- 借鉴 Obsidian Sync 的"冲突文件保留双份,不静默丢弃用户数据"策略——
  同步冲突时永远保留双份让用户手动合并,而不是自动选一份覆盖,这条直接
  影响用户对同步可靠性的信任,值得照抄。
- 借鉴 Notion 的"未同步状态可见"UI 细节(顶栏一个小圆点显示同步进
  行中/已完成/失败)——把同步状态做成用户随时可见的环境感知,而不是藏
  在设置页里,预期能降低"是不是丢数据了"的用户焦虑。
```

## 完成的标准(DoD)

- `<workspace>/docs/competitive-analysis.md` 存在,四段结构完整、顺序
  正确。
- 每一条对标事实(名单里的定位描述、北极星猜测、差异对比里引用的具体数
  据)要么有真实来源标注,要么明确写"未核实/人工提供"——不存在既无来
  源也未标注的断言句。
- 走了路径 B 的报告,文件顶部有材料来源标注(原文见上节步骤 3)。
- 改动是活分支上的真实提交,不是口头描述。

## 常见坑

- **检索工具调用失败就悄悄换成"凭常识写"**:失败了要么重试、要么如实
  切换到路径 B 并标注,不能失败一次就在正文里不标注地混入编造内容。
- **"未核实"标注被当成免责声明滥用**:如果这次真的检索到了,就该标真
  实来源,不能因为省事就统一写"未核实"敷衍了事——那和路径 B 的诚实降
  级不是一回事,是偷懒。
- **把"竞品有的功能"直接抄进可借鉴打法,不说明为什么对本项目有效**:
  每条打法必须落到"为什么对本项目的用户/问题域有效",不是抄一份竞品功
  能清单。
- **北极星猜测和差异定位循环引用、自说自话**:猜测对标的北极星时,依据
  应该来自对标自己的公开信息(定价策略、产品设计侧重、官方披露),不是
  从本项目的立场反推"它应该在乎什么"。
