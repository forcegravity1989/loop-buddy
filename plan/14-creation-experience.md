# 14 · 创建过程体验规范(v1)

日期:2026-07-24 · 状态:grilling 确认「照此落地」,拆票执行中
来源:用户实测创建流后的 grilling(体感三诉求 + 实测两缺陷)。上承 plan/13(GitHub 主体化)——本篇只管**体验维度**;技术偏差的修正是落地注记,不是用户决策。

## 缘起:实测体感台账(2026-07-24)

用户真实跑创建流后报告:

1. **周期判定长期卡住,体感像 mock 数据**——起草卡停在「周期判定 …」很久,无任何过程反馈。
2. **无法退出正在创建的项目**——创建流中途没有任何回主界面项目看板的路,被困。
3. **起草失败原样吐技术黑话且无出路**——`executor failed: Reached maximum budget ($0.5)(subtype=error_max_budget_usd)`,前三相位 ✓ 第四相位挂,失败态卡片上没有任何前进/退出按钮(源码证实:`done` 条件要求 `failed.is_none()`,失败时零按钮渲染)。
4. **GitHub CLI 没有解耦成组件的体感**——真正诉求是**仓平台是一个选择器**(GitHub/GitLab/Gitcode…可选可切),选了谁,创建过程的样式/状态/metadata 就和谁真实映射。今天只有 GitHub 一个选项没关系,但它必须是「选出来的」。

**技术成因(落地注记,已核实,不是用户决策面)**:P1「建项目即建仓」+ plan/13 GitHub 主体化让项目出生即有工作区,而执行器选择规则=「有工作区→真 ClaudeCliExecutor」——创建流的体系起草 workflow 因此意外路由到真 `claude -p`(每相位一次真实调用,经网关,单相位超时上限 30 分钟,预算 $0.5 封顶),违背 plan/13 D8「创建流不跑真 agent」的拍板。修正方向:创建流起草锁回 mock/模板,真起草由标配 Issue 承担(D8 本意)。

## 体验规范(五条,已确认)

1. **永远退得出去**。创建流任何一卡、任何状态(含起草进行中/失败中)都有回主界面项目看板的路;项目已落库的照常留在墙上,回来能续(既有「中断创建可恢复」机制不变)。
2. **后台动作永远有状态回显**。凡超过一瞬的动作(建仓、克隆、起草各相位、标配建单),界面可见:正在干什么、走到哪一相、花了多久、是否在重试——绝不出现静止的「…」让人怀疑是假数据。
3. **失败说人话、给出路**。失败原因翻译成用户语言(「预算到顶,起草没做完」,不是 `subtype=error_max_budget_usd`);当场三出路:重试 / 先用模板继续往下走 / 退出(已完成相位留档不丢)。创建这件事本身永不因网络/预算失败。
4. **仓平台是选择器**。创建入口明确「你在为哪个平台建仓」——今天只有 GitHub 可选,其他平台如实灰置「未接」,绝不假装可用;项目落 provider 字段(双守卫迁移)。选了谁,后续过程与谁真实映射:接入已有仓时回显真实 metadata(描述/可见性/默认分支/最近推送),不是干巴巴一个仓名。平台抽象不超前设计——第二个平台真要接的时候再抽缝(用户明确「别想复杂了」)。
5. **快慢分层**。创建流本身秒级利落(一气呵成);真正耗时的真 agent 工作(找指标、竞品分析)以标配 Issue 形态在看板后台进行,进度在卡片可见——慢活不堵创建的路。
