# 13 · 创建流与 GitHub 主体化:入口、Issue 验收环、指标管线(v1)

日期:2026-07-23 · 状态:已过 grilling(12 项拍板),spec 定稿,拆票执行中
来源:/grill-me 盘问全记录(本文即共识存档)。上承 `docs/superpowers/specs/2026-07-22-github-centric-onboarding-design.md`(入口部分**继续有效**,本文仅推翻其非目标中两条,见 D12)。

## 问题陈述

今天在 BW 里建一个项目,得到的是一个本地目录和一套模板指标:项目的交付件(代码、Spec、原型)和管理体系(issue、指标)与我真实的 GitHub 仓是两张皮。issue 是 BW 私有的,干完的活不落在任何外部可见的地方;北极星指标是创建流模板填的,没有一条真的接着数据;看板要么灰、要么靠手填。管理体系和真实世界脱钩——「健康难造假」的灯照不到产品本身。

## 解决方案

**建项目=开(或接入)一个 GitHub 仓。** 产品信息(代码/Spec/原型/issue/指标定义)的正本住在仓里,过程信息(会话/run/记账/信号/交棒)住在 BW——BW 是操作台+信息转化层。BW 建单即真开 GitHub issue;活的交付物一律 PR 进仓;**验收=人 merge**,issue 关闭是 merge 的后果,执行器只会提 PR、永远不能 merge。创建流自动建三张标配 Issue(竞品分析→找指标→绑数据)挂上标配 Skill,末卡问一句「立即让队友开工第一件?」。指标定义以机读文件进仓为正本,标配 Cron 采集器把 GitHub 数据变成 append-only 观测,看板每盏灯可溯源——无数据=Unknown,绝不假装绿。

## User Stories

1. 作为构建者,我希望建项目时直接新建一个 GitHub 私有仓(或接入已有仓),让项目从第一天起就有真实交付载体。
2. 作为构建者,我希望项目显示名可用中文、仓名(slug)单独预览可改,不被事后静默转写。
3. 作为构建者,我希望新建仓失败时软降级为本地仓并如实提示,创建永不因网络失败。
4. 作为构建者,我希望接入已有仓失败时项目照常建立、仓位留空并如实提示,而不是伪造一个无关空仓充数。
5. 作为构建者,我希望在 BW 建的每张 Issue 都真实存在于 GitHub 仓的 Issues 里,任何外部工具都能看到我的工作队列。
6. 作为构建者,我希望 AI 队友干完活只会提 PR、永远不能 merge,验收权始终在我手里。
7. 作为构建者,我希望 merge 一个 PR 后对应 issue 随之关闭,「验收」只需一个动作、不用两头点。
8. 作为构建者,我希望非代码活(竞品分析、复盘、运维留痕)也以文档 PR 交付,所有交付物进仓、历史即 git 历史。
9. 作为构建者,我希望创建流结束时自动建好「竞品分析→找指标→绑数据」三张标配 Issue,不用自己发明项目起手式。
10. 作为构建者,我希望创建流末卡问我「立即让队友开工第一件?」,勾了就自动开工竞品分析——授权显式、开工零摩擦。
11. 作为构建者,我希望竞品分析活产出对标名单/各家北极星猜测/差异定位/可借鉴打法的报告进仓,「知道对标谁」有实物可查。
12. 作为构建者,我希望执行器的联网检索能力先探活,不通时如实降级为「我喂材料+队友整理」,报告绝不由幻觉填充。
13. 作为构建者,我希望找指标 Skill 起草北极星/滞后/引领三层指标且**每条附采集方案**,指标是「对的」而不是「采得到的」。
14. 作为构建者,我希望指标定义正本是仓里的机读文件、改指标走 PR,指标变更和代码变更一样过验收门。
15. 作为构建者,我希望绑不上数据的指标如实显示 Unknown 并附点亮它的最便宜路径,看板永不假装绿。
16. 作为构建者,我希望标配 Cron 采集任务每天把 GitHub 数据拉成观测、外加「立即同步」按钮,健康灯由真实数据点亮、不靠我勤快。
17. 作为构建者,我希望手填的指标值带「手填」徽记,回看时分得清哪些数字是机器采的。
18. 作为构建者,我希望看板上每个数字都能用 sqlite3 独立查证到观测来源——报告不代答,读回为证。
19. 作为构建者,我希望原型设计在外部设计系统里做、成品用其导出能力落进仓走 PR,用最好的画布而产物仍归档在仓。
20. 作为构建者,我希望 BW 登记外部产物时标注来源,产物清单如实反映哪些是外部系统产出。
21. 作为 AI 队友,我希望接到标配 Issue 时自动注入对应 Skill(竞品分析/找指标/绑数据),同类活越干越熟(复利链既有机制)。
22. 作为构建者,我希望 GitHub 上不经 BW 发生的状态变化被如实采集展示、而不是被 BW 反向改写,BW 与 GitHub 永不打架。
23. 作为构建者,我希望旧项目(无 GitHub 仓)打开一切如常,存量库绝不因升级而崩(双守卫迁移铁律)。
24. 作为构建者,我希望创建流本身依旧乐观推进、秒级完成,网络抖动永远卡不住建项目。

## 实现拍板(Implementation Decisions)

grilling 十二项拍板,全部已确认:

- **D1 信息切分**:产品信息(代码/Spec/原型/issue/指标定义)正本在 GitHub 仓;过程信息(会话/run/记账/信号/交棒)在 BW SQLite。BW=操作台+信息转化层。
- **D2 Issue 身份**:BW 的 issue 就是 GitHub 的 issue(issue 号即身份)。BW 建单=经 GitHub Connector(gh shell-out)真开 issue;BW 上的动作实时映射为 GitHub 操作。存量无仓项目的 Issue 保持本地身份,如实留白、不假装迁移。
- **D3 验收=merge**:执行器只提 PR、永不 merge;merge 动作只存在于人手;issue 关闭是 merge 的后果。「Done 永不自动」铁律的锚点从「BW 点 Done」迁至「人点 merge」;「评审中」由「存在开放 PR」派生。
- **D4 全活 PR 化(v1)**:每件活的交付物都以 PR 进仓,非代码活交付文档文件。先立起来,重了再松。
- **D5 指标正本机读**:仓里一份结构化文件(`.bw/metrics.toml` 形态)承载每条指标的定义+层级(北极星/滞后/引领)+采集方案;merge 后同步进 SQLite 作缓存;观测/信号仍走既有 append-only + `Derived<Signal>` 派生链,一点不松。
- **D6 先对后亮**:找指标 Skill 起草「对的」指标,不受当前采集能力约束;每条必须附采集方案(GitHub 查询 / Connector / BW 自记账 / 手填徽记);绑不上的如实 Unknown,绑数据 Skill 给出点亮的最便宜路径。北极星绝不为「采得到」退化成 commit 数。
- **D7 采集器**:22 号设计非目标中的「墙 C 后置」被推翻——创建流自动建标配 Cron 采集任务(每日 pull GitHub→观测)+手动「立即同步」。采集是观测不是活,自动跑不犯「Done 永不自动」的忌。
- **D8 标配 Issue 三件套**:创建流不跑真 agent,自动建竞品分析→找指标→绑数据三张标配 Issue(竞品分析报告是找指标的输入),挂对应 Skill,走现成 RunIssue→提 PR→人 merge 机器。末卡显式勾选「立即让队友开工第一件?」,勾了自动开工竞品分析那张。
- **D9 竞品分析 v1**:单次研究活,报告 PR 进仓;执行器检索能力先探活,不通则如实降级为「人喂材料+agent 整理」;多轮对抗式评审留给 plan/12 的 Workflow 机制成熟后。
- **D10 原型对接**:出口进仓、零集成。设计在外部系统界面里做(允许另一个工作界面),用对方导出能力落进仓走 PR;BW 只登记 artifact(仓内路径+来源标注)。原型设计不是创建流标配 Issue,是原型阶段的正活。
- **D11 merge 门守在 BW 层**:不做 bot 账号/分支保护等 GitHub 侧硬隔离(私有仓分支保护要付费档,且同身份无法自审 PR)。执行器权限配置里禁 merge,merge 按钮只在 BW 人手界面。
- **D12 入口沿用 22 号设计**:Repo 卡(新建/接入)、slug 预览、`gh` shell-out、软降级、`github_remote` 双守卫迁移全部照旧实施;仅推翻其非目标中「墙 C 后置」与「github-repo connector 不接真探针」两条。

**模块落点(概念级)**:GitHub shell-out 能力扩展在 bw-engine 的 github 模块(issue/PR/统计三组子命令);命令与守卫全部在 bw-app 命令层;schema 变更(issue 关联 GitHub issue 号与 PR 号、metric 定义缓存的来源标注)一律双守卫迁移;UI 改动只进 app-desktop。内核 crate 仍禁 UI 依赖、wasm32 保活。

## 测试拍板(Testing Decisions)

- **纪律照旧**:不写单元测试。行为正确性 = E2E(深链启动 stderr 证明 + sqlite 读回)+ 门禁全过 + `/code-review`。好测试只验外部行为:dispatch 命令→读回 DB/工作区,不摸内部实现。
- **测试缝**:最高缝=bw-app 的 Command/Event 总线(既有 E2E 主战场,`real_demo`/`practice_aihot` 先例)。本特性**不新开缝**。
- **gh 副作用的验证分层**:管线验证用 PATH 前置的 stub gh 脚本(输出自我标注【mock】,mock 必须自我标注铁律);**真实账号 E2E 单独成票**(会在用户 GitHub 账号上开真仓/真 issue,执行前报告并等确认,验证后问是否清理)。
- **UI 编译过即可**,行为在命令层+E2E 兜底(既有纪律,如实不假装 UI 测试)。

## 范围外(Out of Scope)

- GitHub OAuth/token 管理(完全沿用本机 `gh auth`)
- bot 账号、分支保护等 GitHub 侧硬隔离(D11)
- 多仓/mono-repo 项目
- issue/PR/基础统计之外的 GitHub 面(Discussions、Projects、Actions 配置)
- 双向同步的冲突解决——BW 永不反向改写 GitHub,漂移只采集、只展示
- 对抗式多轮竞品分析 Workflow(等 plan/12 机制成熟)
- 外部设计系统的 Connector/API 集成、画布嵌入(零集成,只认落进仓的文件)
- 存量本地 Issue 批量迁移到 GitHub(留白如实标注)

## 附注(Further Notes)

- **四批次**:①入口(按 22 号既有 2012 行实现计划先行)→ ②Issue↔GitHub 映射+PR 验收环(最大块,标配 Issue 与采集器都压在它上面)→ ③指标正本+采集器+看板 → ④标配 Issue 三件套+Skill 内容+问一句就跑。
- **并行流风险**:plan/12 票流(T1-T11)正在另一 worktree 执行,与本流在 bw-app/bw-store 热点文件上有合流冲突风险;本流串行执行、每票独立 commit,合流时以先落 main 者为准。
- **留白如实**:外部设计系统具体产品名待核(零集成下不影响决策,落地时核对导出能力);执行器联网检索未探活,是竞品分析 v1 的硬前提,探活结果决定 v1 还是降级形态。
- **执行分派**:多数票 Sonnet 5(max);PR 验收环与采集器两票 Opus 4.8;真账号验收、监理与 code-review 由 Fable 5(主 agent)。
